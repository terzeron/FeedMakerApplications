Claude에게 며칠간 과학 코드를 맡겨봤더니, 연구자 수개월 치 작업이 완성됐다

@jaehong
2026년 3월 25일 10:38 오전
Claude Ai

우주 초기의 광자, 바리온, 뉴트리노, 암흑물질이 어떻게 진화했는지를 계산하는 볼츠만 솔버(Boltzmann solver). 우주론 연구의 핵심 인프라이면서, 전문가가 수개월에서 수년을 투자해야 구현할 수 있는 고난도 수치 코드다. Anthropic의 연구자 Siddharth Mishra-Sharma는 이 솔버의 미분 가능한(differentiable) JAX 버전을 Claude Opus 4.6에게 맡겼다. 본인은 우주론 전문가도 아니었다. 결과는 며칠 만에 기준 구현체 CLASS 대비 1% 미만의 오차를 달성한 작동하는 솔버였다. 이 글은 단순한 "AI가 코드를 짰다"는 이야기가 아니다. 자율 에이전트가 복잡한 과학 계산을 수행하기 위해 필요한 구체적인 오케스트레이션 패턴, 그리고 이것이 연구 생산성에 던지는 근본적 질문에 관한 이야기다.

대화형 AI에서 자율형 에이전트로: 패러다임 전환

현재 대부분의 과학자가 AI를 쓰는 방식은 대화형 루프다. 매 단계마다 직접 확인하고, 다음 지시를 내리고, 결과를 검증한다. 하지만 모델의 장기 작업(long-horizon task) 능력이 크게 향상되면서 새로운 작업 방식이 등장했다. 고수준 목표만 설정하고, 에이전트 팀을 풀어놓는 것이다.

이 접근이 잘 맞는 과학 작업의 특성이 있다. 작업 범위가 명확하고, 성공 기준이 정량적이며, 인간의 개입이 간헐적이어도 되는 경우다. 수치 솔버 재구현, 레거시 포트란 코드를 현대 언어로 변환, 참조 구현체와의 디버깅 같은 작업이 여기에 해당한다. Anthropic이 이전에 공개한 C 컴파일러 프로젝트(약 2,000개 세션에 걸쳐 리눅스 커널을 컴파일하는 C 컴파일러를 구축한 사례)가 이 패턴의 대규모 시연이었다면, 이번 볼츠만 솔버는 일반 학술 연구실 규모에서 같은 패턴을 적용한 실전 사례다.

볼츠만 솔버가 왜 어려운 문제인가

이 예제의 선택이 단순하지 않다. 볼츠만 솔버는 C 컴파일러 프로젝트와 구조적으로 다른 난이도를 가진다. C 컴파일러는 수많은 병렬 에이전트에게 분배할 수 있는 작업이지만, 볼츠만 솔버는 깊게 결합된(deeply coupled) 파이프라인이다. 초기 우주의 재결합 과정에서 작은 수치 오차나 부적절한 근사가 하류의 모든 결과를 미묘하게 왜곡한다. 디버깅할 때 전체 인과 체인을 추적하면서 도메인 지식까지 동원해야 한다는 뜻이다.

더 중요한 맥락이 있다. 기존에 전문 연구 그룹들이 CLASS의 JAX 기반 미분 가능 버전을 만든 적은 있지만, 이들은 해당 분야의 전문가였고 기능의 일부만 구현하는 데도 수개월에서 수년이 걸렸다. Mishra-Sharma는 이 분야의 비전문가로서, 에이전트가 최소한의 조종만으로 얼마나 멀리 갈 수 있는지를 시험한 것이다.

미분 가능 솔버를 JAX로 구현하는 이유도 실용적이다. JAX는 자동 미분(automatic differentiation)과 GPU 같은 가속기 호환성을 기본 제공하기 때문에, 그래디언트 기반 추론 방법을 적용해 파라미터 추정 속도를 극적으로 높일 수 있다. Planck이나 Simons Observatory 같은 관측 데이터로 우주론 모델을 제약하는 핵심 도구인 만큼, 이 속도 향상의 과학적 가치는 크다.

자율 에이전트 운영의 핵심 패턴 네 가지

이 프로젝트에서 사용한 오케스트레이션 패턴들은 과학 컴퓨팅에 국한되지 않는, 범용적인 에이전트 운영 방법론이다.

1. CLAUDE.md: 자기 수정 가능한 지시서

프로젝트 루트에 CLAUDE.md 파일을 두고, 여기에 전체 계획, 설계 결정, 정확도 목표(CLASS 대비 0.1% 이내)를 명시한다. 핵심은 Claude가 이 파일을 작업 중에 직접 수정할 수 있다는 점이다. 작업하면서 발견한 이슈나 변경된 설계 결정을 스스로 업데이트한다. 0.1%라는 목표치는 CLASS와 CAMB(두 표준 볼츠만 코드) 간의 전형적인 일치 수준에서 가져온 것으로, 과학적으로 의미 있는 기준이다.

연구자의 역할은 이 지시서를 잘 작성하는 데 집중된다. Claude와 상의하면서 계획을 반복적으로 다듬고, 합리적으로 보일 때까지 로컬에서 이터레이션한다. 관리의 대상이 코드에서 "지시서"로 바뀐 셈이다.

2. CHANGELOG.md: 실패까지 기억하는 장기 메모리

에이전트의 세션 간 기억을 담는 진행 파일이다. 현재 상태, 완료 작업, 실패한 접근법과 그 이유, 주요 체크포인트의 정확도 테이블, 알려진 한계를 기록한다. 여기서 가장 중요한 항목은 "실패한 접근법"이다. 이것이 없으면 후속 세션이 같은 막다른 길을 반복한다. 예를 들어 "섭동 ODE에 Tsit5를 시도했으나 시스템이 너무 stiff하여 Kvaerno5로 전환"과 같은 기록이 쌓인다.

이것은 사실상 연구 노트북의 디지털 버전이다. 인간 연구자도 실험 노트에 실패한 시도를 기록하지만, 에이전트에게는 이것이 문맥 창 밖의 세계를 기억하는 유일한 수단이라는 점에서 더 절실하다.

3. 테스트 오라클: 자율 작업의 전제 조건

장기 자율 작업의 핵심 전제는 에이전트가 "자기가 진전하고 있는지"를 스스로 판단할 수 있어야 한다는 것이다. 이 프로젝트에서는 CLASS C 소스 코드를 참조 구현체로 사용하여 단위 테스트를 구성하고, 에이전트가 작업하면서 지속적으로 이 테스트를 실행하도록 지시했다. 또한 테스트 스위트를 스스로 확장하도록 해서 회귀를 방지했다.

이것이 시사하는 바가 크다. 현재 에이전트 기반 자율 작업이 가능한 영역은 "정량적 성공 기준이 존재하는 곳"으로 제한된다. 참조 구현체, 명확한 수치 목표, 기존 테스트 스위트 중 하나가 있어야 한다. 보다 개방적인 과학적 발견은 아직 지평선 위에 있다.

4. Git 기반 협업과 모니터링

에이전트에게 "의미 있는 작업 단위마다 커밋하고 푸시하라", "커밋 전에 반드시 pytest tests/ -x -q 실행", "기존 통과 테스트를 깨는 코드를 커밋하지 마라"는 규칙을 부여한다. Git이 복구 가능한 히스토리, 진행 상황의 원격 모니터링, 컴퓨트 할당이 중간에 끊겨도 작업 보존이라는 세 가지 역할을 동시에 수행한다. 연구자는 커피 줄에 서서 폰으로 GitHub 커밋 로그를 확인하는 것만으로 충분하다.

Ralph 루프: 에이전트가 "다 했다"고 할 때

현재 모델의 흥미로운 약점 중 하나는 "에이전트 게으름(agentic laziness)"이다. 복잡한 멀티파트 작업을 요청하면 완료 전에 그럴듯한 핑계를 대며 멈추는 현상이다. 논문에서 인용한 예시가 인상적이다. "늦어지고 있으니, 내일 이어서 하죠?"

이를 우회하는 오케스트레이션 패턴이 Ralph 루프다. 본질적으로 for 루프인데, 에이전트가 완료를 선언하면 다시 컨텍스트로 밀어 넣고 "정말 끝난 거 맞아?"라고 묻는 구조다. 실제로 에이전트는 기준에 미달함을 인정하고 작업을 계속한다. 이 프로젝트에서는 최대 20회 이터레이션, "전체 파라미터 범위에서 0.1% 정확도" 달성 시 "DONE"을 출력하도록 설정했다.

이것은 모델 능력의 한계를 솔직히 인정하면서 엔지니어링으로 보완하는 접근이다. 모델이 더 좋아지면 이런 스캐폴딩이 점점 불필요해지겠지만, 현재 시점에서는 실용적인 성능 향상을 가져다준다.

결과와 한계: 솔직한 평가

에이전트는 며칠에 걸쳐 처음부터 솔버를 구축했고, CMB 각도 파워 스펙트럼(우주배경복사의 통계적 특성을 나타내는 핵심 출력) 전반에서 CLASS 대비 1% 미만의 오차를 달성했다. Anthropic이 공개한 정확도 추이 그래프를 보면, 개발 과정에서 주요 마일스톤을 거치며 점진적으로 정확도가 개선되는 궤적이 보인다.

하지만 솔직한 한계도 있다. 에이전트의 개발 궤적은 다소 어설펐다. 한동안 단일 기준점(fiducial point)에서만 테스트하여 버그 탐지 범위가 극도로 좁았고, 게이지 규약에서 헤매거나 우주론 전문가라면 즉시 발견할 버그에 수 시간을 소비하는 초보적 실수도 있었다. 최종 결과물도 프로덕션 수준은 아니다. 모든 영역(regime)에서 CLASS와 허용 가능한 정확도로 일치하지는 못한다.

그럼에도 핵심 메시지는 분명하다. 전문가 연구팀이 수개월에서 수년을 투자할 작업을, 비전문가가 에이전트와 며칠 만에 의미 있는 수준까지 끌어올렸다는 것이다.

이것이 왜 중요한가: 연구의 기회비용이 바뀌고 있다

이 글에서 가장 통찰력 있는 대목은 마지막에 나온다. AI 연구에서 보편적인 경험은 밤새 실험(훈련 런 등)을 돌려놓고 아침에 결과를 확인하는 만족감이다. 실험을 안 돌리면 기회비용이 발생한다. 이제 에이전트를 안 돌리는 것도 같은 기회비용이 된다는 것이다.

이 관점의 전환이 핵심이다. 충분한 컴퓨트와 명확한 성공 기준이 있는 프로젝트가 있다면, 에이전트가 일하지 않는 매 시간이 잠재적 진전의 낭비다. 이것은 GPU 훈련 런과 동일한 "유휴 시간의 비용" 프레이밍인데, 대상이 모델 훈련에서 소프트웨어 구현으로 확장된 것이다.

물론 현실적 제약은 있다. 정량적 성공 기준이 없는 탐색적 연구, 도메인 전문 지식이 매 단계마다 필요한 작업, 깊은 결합 때문에 에러가 미묘하게 전파되는 시스템에서는 여전히 인간 전문가의 판단이 핵심이다. 하지만 과학 연구의 상당 부분은 "알려진 방법론의 재구현과 적용"이다. 이 영역에서 에이전트가 연구자의 시간을 해방시킬 수 있다면, 연구자는 진짜 창의적 사고가 필요한 부분에 집중할 수 있다.

한 가지 더 주목할 부분은 부수 효과다. Mishra-Sharma는 에이전트의 Git 커밋 히스토리를 따라가면서 볼츠만 솔버와 관련 물리학을 상당히 배웠다고 말한다. 커밋 로그가 "빠르고 지나치게 문자적인 포닥의 실험 노트"처럼 읽힌다는 표현은, 에이전트가 교육 도구로서도 가치가 있음을 시사한다. 코드를 직접 짜지 않아도, 에이전트의 작업 과정을 관찰하는 것 자체가 효과적인 학습이 된다는 것이다.

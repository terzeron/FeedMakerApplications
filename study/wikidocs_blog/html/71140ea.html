ASP.NET Blazor에서 JavaScript와의 양방향 통신 모험기

@dotlynee
2025년 4월 2일 8:17 오후
Jsinterop Asp.net Javascript Blazor Typescript Vite

이 글에는 타입스크립트로 작성한 자바스크립트 코드를 Vite로 빌드한 뒤 Blazor와 양방향 통신을 시도한 닷넷 초보의 모험이 담겨져 있습니다. Github 레포지터리

💎 모험에서 얻은 보상

1. 닷린이의 첫 발걸음: Dockview를 써보고 싶어요... 현기증 난다구욧!

안녕하세요! 저는 닷넷 생후 1개월 된, 커여운 햄스터 닷린이입니다. 이번 모험은 Blazor에서 JavaScript와의 양방향 통신을 구현해보려는 도전기입니다. 목표는 Dockview 라는 도킹 레이아웃 매니저를 Blazor에서 사용하기 위해 JavaScript 라이브러리를 활용하는 것이었죠.

혹시 DockView를 모르신다구요?! Dockview는 이런 멋진 도킹 레이아웃 매니져입니다. 이걸 Blazor 컴포넌트와 같이 사용하면 얼마나 멋진 일일까요? 😍

하지만, 이 여정은 순탄치 않았습니다. 일단 저는 닷넷 왕초보 닷린이입니다. ASP.NET, Blazor 모두 기초만 끄적거려봤을 뿐이죠. 😭 일단 저 Dockview 는 바닐라 JS로 만들어진 JavaScript 라이브러리입니다. 직접 DOM 을 핸들링하는데 여기다가 Blazor 컴포넌트를 곁들여야 하죠. 도대체 어디서부터 시작해야 할지 막막했습니다.

하지만 저에겐 퍼플렉시티님이 있었기에, 닥치고 자바스크립트와이 랑데뷰를 시도해봤습니다. 비록 삽질과 멘붕의 연속이었지만, 그 과정에서 얻은 깨달음을 공유하려 합니다 🙏

2. 프로젝트 구조 세우기: 닷린이의 첫 삽질

처음에는 퍼플렉시티에게 "해죠~" 하며 Blazor JSInterop 연습용 프로젝트를 만드는 방법을 문의했어요.

다음 명령으로 Blazor Web App 을 만들라고 하더군요.

dotnet new blazor -o BlazorJSInteropDemo

그 결과, 만들어진 기본적인 프로젝트 구조는 아래와 같았습니다:

BlazorJSInteropDemo/
│   BlazorJSInteropDemo.csproj
│   Program.cs
├───Components
│   │   App.razor
│   │   Routes.razor
│   │   _Imports.razor
│   ├───Layout
│   └───Pages
├───Properties
│       launchSettings.json
└───wwwroot
    └───lib
        └───bootstrap
            └───dist
                ├───css
                └───js

여기서 wwwroot/js/site.js 파일에 자바스크립트를 직접 작성해야 했는데, 이 방식은 저에게 너무 불편했습니다. 저는 IDE의 도움을 받아 타입스크립트로 작업하고 싶었거든요! 그래서 다음과 같은 구조로 변경하고 싶었습니다:

BlazorJSInteropDemo/
├───Client
│   └───src
│       ├─ package.json
│       ├─ vite.config.ts
│       └─ site.ts - - .
├───Components         :
├───Properties         :
└───wwwroot            :
    ├───js             :
    │   └───dist       v
    │           blazor-interop.js
    └───lib

타입스크립트 파일(site.ts)을 작성하고, 이를 Vite 빌드 매니저를 통해 wwwroot/js/dist/site.js로 변환하는 구조를 만들기로 했습니다.

3. 첫 번째 난관: Vite 빌드 설정 삽질기

타입스크립트를 사용하려면 vite.config.ts 파일을 설정해야 했습니다. 하지만 인터넷에는 적절한 레시피가 거의 없었고, 퍼플렉시티님도 명확한 답을 주지 못했어요. 😭 결국 시행착오 끝에 다음과 같은 설정을 완성했습니다:

import { defineConfig } from 'vite';

export default defineConfig({
  build: {
    lib: {
      entry: 'src/site.ts', // 진입점 파일
      formats: ['es'],      // ES 모듈로 출력
      fileName: () => 'blazor-interop.js',
    },
    rollupOptions: {
      output: {
        exports: 'named',  // named exports 유지
      },
    },
    minify: true,
    outDir: '../wwwroot/js/dist',
    emptyOutDir: false,
  },
});

이제 타입스크립트로 작성한 코드가 wwwroot/js/dist/blazor-interop.js로 빌드되었고, Blazor에서 사용할 수 있게 되었어요! 🎉

참고로, site.ts 는 이렇게 생겼답니다.

import { DotNet } from "@microsoft/dotnet-js-interop";

export function AddTableRow(
    tableBody: HTMLElement,
    name: string,
    city: string,
    dotNetRef: DotNet.DotNetObject // .NET 객체 참조
): void {
    try {
        const row = document.createElement('tr');
        row.innerHTML = `<td>${name}</td><td>${city}</td>`;
        tableBody.appendChild(row);

        // .NET 메서드 호출 (1초 후 비동기 실행)
        setTimeout(() => {
            dotNetRef.invokeMethodAsync('OnRowAdded', name, city);
        }, 1000);

    } catch (error) {
        dotNetRef.invokeMethodAsync('OnJSError', (error as Error).message);
    }
}

4. 두 번째 함정: 퍼플렉시티의 배신(?)

퍼플렉시티님은 HandleJsCallback 메서드를 다음처럼 작성하라고 알려줬습니다:

[JSInvokable("OnRowAdded")]
public static void HandleJsCallback(string name, string city)
{
    Console.WriteLine($"JS에서 전달된 데이터: {name}, {city}");
}

하지만 이 코드는 제대로 동작하지 않았습니다! 😱 왜냐하면 static 메서드는 Blazor 컴포넌트의 상태를 변경할 수 없기 때문이에요.

햄스터의 깨달음

퍼플렉시티님도 완벽하지 않다는 사실을 깨달았어요... 😅

햄스터의 해결책

static 키워드를 제거하고 인스턴스 메서드로 변경했습니다:

[JSInvokable("OnRowAdded")]
public void HandleJsCallback(string name, string city)
{
    Console.WriteLine($"JS에서 전달된 데이터: {name}, {city}");
    StateHasChanged(); // UI 갱신 가능!
}

이제야 제대로 동작하네요! 퍼플렉시티님, 다음엔 더 정확히 알려주세요~ 😊

5. DotNetObjectReference 관리 삽질기

다음 문제는 DotNetObjectReference 관리였어요. 참조 해제를 제대로 처리하지 않아 메모리 누수가 발생했습니다.

햄스터의 해결책

컴포넌트 생명주기와 참조 수명을 일치시키는 것이 핵심이에요:

private DotNetObjectReference<DOMInterop>? _dotNetRef;

protected override void OnInitialized()
{
    _dotNetRef = DotNetObjectReference.Create(this);
}

public async ValueTask DisposeAsync()
{
    _dotNetRef?.Dispose(); // 참조 해제!
}

6. 최적화와 성능 개선

양방향 통신 구현 후 성능 문제가 발생했어요. JS Interop 호출이 많아질수록 성능 저하가 눈에 띄었죠.

햄스터의 해결책

  1. Batch 처리:
    여러 데이터를 한 번에 처리하도록 TypeScript 함수를 수정했어요:
export function AddMultipleRows(tableBody: HTMLElement, data: Array<[string, string]>): void {
    data.forEach(([name, city]) => {
        const row = document.createElement('tr');
        row.innerHTML = `<td>${name}</td><td>${city}</td>`;
        tableBody.appendChild(row);
    });
}
  1. Lazy Initialization:
    JS 모듈 로드를 지연시켜 필요할 때만 초기화:
private readonly Lazy<Task<IJSObjectReference>> _jsModuleTask;

public DOMInterop(IJSRuntime jsRuntime)
{
    _jsModuleTask = new(() => jsRuntime.InvokeAsync<IJSObjectReference>("import", "./js/dist/blazor-interop.js").AsTask());
}

7. 결론: 삽질 끝에 얻은 보물

이번 모험을 통해 얻은 교훈은 다음과 같습니다:

  1. 타입스크립트 빌드 설정은 Vite와 Rollup 옵션을 활용해 export를 유지해야 합니다.
  2. 퍼플렉시티님도 무조건 신뢰하지 말고 문제를 직접 디버깅하세요.
  3. DotNetObjectReference는 반드시 생명주기와 일치시키고 Dispose()로 해제하세요.
  4. 성능 최적화를 위해 Batch 처리와 Lazy Initialization을 적극 활용하세요.

햄스터 닷린이는 앞으로도 삽질하며 성장할 예정입니다! 여러분도 함께 삽질하며 성장하시길 바라요! 🐹

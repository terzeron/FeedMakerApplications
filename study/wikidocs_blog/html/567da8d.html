async_await 내부에서 실제로 어떤 일이 일어나는가

@lukewire
2026년 2월 27일 6:09 오전
Asyncawait Javascript Csharp 비동기프로그래밍

await를 쓰면 "기다린다"는 건 알겠는데, 그 사이에 스레드는 뭘 하는가.

배경

C#에서 async/await를 쓰다가 JavaScript로 넘어오면 묘하게 헷갈린다.
동작은 비슷해 보이는데, 내부 구조는 꽤 다르다.

특히 "JS는 싱글 스레드인데 어떻게 비동기가 되는가"라는 질문에
setTimeout, Promise, await가 뒤섞이면 더 헷갈린다.

이 글은 그 내부를 구체적으로 파고든다.

먼저 알아야 할 것: 자바스크립트 실행 모델

JS 엔진은 싱글 스레드다.
동시에 두 코드가 실행되는 일은 없다.

대신 세 가지 구조로 돌아간다.

  • Call Stack: 현재 실행 중인 함수들의 스택
  • Microtask Queue: Promise 콜백이 쌓이는 큐
  • Macrotask Queue: setTimeout, I/O 콜백 등이 쌓이는 큐

이벤트 루프의 동작 순서는 이렇다.

Call Stack이 비면
→ Microtask Queue를 전부 소진
→ 그 다음 Macrotask 하나 처리
→ 반복

이게 핵심이다.
await는 이 구조 위에서 동작한다.

async 함수는 무엇인가

async 키워드를 붙이면 함수는 항상 Promise를 반환한다.

async function foo() {
  return 42;
}

// 위와 완전히 같다
function foo() {
  return Promise.resolve(42);
}

리턴값이 자동으로 Promise로 감싸진다.
await 없이 async만 써도 동작은 동일하다.

await를 만나면 무슨 일이 생기나

async function fetchData() {
  console.log("A");
  const data = await fetch("/api/data");
  console.log("B");
  return data;
}

fetchData();
console.log("C");

출력 순서는 A → C → B다.

await를 만나는 순간 다음 일이 일어난다.

  1. fetch()를 호출하고 Promise를 받는다
  2. 현재 함수의 실행을 일시 중단한다
  3. 콜 스택에서 빠진다
  4. 제어권이 호출자에게 돌아간다
  5. console.log("C")가 실행된다
  6. fetch()가 완료되면 그 콜백이 Microtask Queue에 쌓인다
  7. 콜 스택이 비면, 루프가 Microtask를 꺼내 console.log("B")부터 재개한다

내부 구조: 상태 머신

C#을 써봤다면 익숙한 개념이다.
JS 엔진은 async 함수를 컴파일할 때 상태 머신(State Machine) 으로 변환한다.

async function example() {
  const a = await step1();  // 상태 0
  const b = await step2(a); // 상태 1
  return b;                 // 상태 2
}

엔진 내부에서는 대략 이렇게 변환된다.

function example() {
  let state = 0;
  let a, b;

  function resume(value) {
    switch (state) {
      case 0:
        state = 1;
        return step1().then(resume);

      case 1:
        a = value;
        state = 2;
        return step2(a).then(resume);

      case 2:
        b = value;
        return Promise.resolve(b);
    }
  }

  return resume();
}

실제 엔진 코드는 이보다 훨씬 복잡하지만, 핵심 구조는 같다.
await 지점마다 상태가 생기고, Promise가 완료되면 해당 상태부터 재개된다.

C#의 async/await 컴파일 결과물과 거의 같은 패턴이다.

Microtask vs Macrotask: 왜 구분이 중요한가

console.log("1");

setTimeout(() => console.log("2"), 0); // Macrotask

Promise.resolve().then(() => console.log("3")); // Microtask

console.log("4");

출력 순서: 1 → 4 → 3 → 2

setTimeout(..., 0)이라도 Macrotask이기 때문에
Promise의 .then() 보다 늦게 실행된다.

await 이후의 코드는 Microtask Queue에 들어가므로
setTimeout 계열보다 항상 먼저 실행된다.

주의할 점

await는 "블로킹"이 아니다

async function slow() {
  await delay(5000); // 5초
  return "done";
}

slow();
console.log("이게 먼저 찍힌다"); // ✅ 블로킹 없음

await는 현재 함수만 일시 중단한다.
전체 스레드를 멈추는 게 아니다.

Promise를 반환하지 않는 값도 await할 수 있다

const x = await 42; // 그냥 42가 된다

await는 내부적으로 Promise.resolve()로 감싸기 때문에
일반 값을 넣어도 에러 없이 동작한다.
다만 의미가 없는 코드다.

에러 처리

async function example() {
  try {
    const data = await fetch("/api"); // ❌ 실패 가능
    return data;
  } catch (e) {
    console.error(e);
  }
}

await 표현식에서 Promise가 reject되면 throw처럼 동작한다.
try/catch로 잡을 수 있다.

.catch()로 처리하지 않은 rejected Promise는
UnhandledPromiseRejection으로 떨어진다.

C#과 비교

C# async/await JS async/await
스레드 모델 멀티 스레드, ThreadPool 활용 싱글 스레드, 이벤트 루프
상태 머신 컴파일 타임에 생성 엔진이 런타임에 처리
중단 후 재개 위치 같은 스레드 or 다른 스레드 항상 같은 스레드
동기화 컨텍스트 SynchronizationContext 존재 없음 (단일 컨텍스트)

C#에서는 await 이후 다른 스레드에서 재개될 수 있어서
UI 스레드 접근 시 주의가 필요하다.
JS는 항상 같은 스레드이므로 그런 문제가 없다.

마무리

await 는 함수 실행을 일시 중단하고 콜 스택에서 빠진다.
Promise가 완료되면 Microtask Queue를 통해 재개된다.
엔진 내부에서는 상태 머신으로 변환되어 동작한다.

"기다린다"는 표현보다는 "잠시 양보하고, 완료 신호를 받으면 돌아온다"가 더 정확하다.

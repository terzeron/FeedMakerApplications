AWS VPC와 서브넷 구성의 필요성

@develofun
2025년 5월 20일 10:18 오전
Serverless 서버리스 Aws 
AWS에 API 서버를 구축하기 이전에 네트워크 구성을 어떻게 해야할지 몰랐는데 
기본적인 구성이 VPC와 서브넷 구성이 필요하다 하여 그에 대한 서치 부분을 정리해봤습니다.

VPC(Virtual Private Cloud)와 서브넷을 구성해야 하는 이유는 네트워크 격리와 보안을 위한 필수적인 인프라 요소이기 때문입니다. 특히 server-to-server 통신만 허용하고 퍼블릭 접근을 제한하려는 목적에 부합합니다.

VPC 구성이 필요한 주요 이유

1. 네트워크 격리

VPC는 AWS 클라우드 내에서 논리적으로 격리된 가상 네트워크를 제공합니다. 이를 통해 다른 AWS 고객의 리소스와 완전히 분리된 환경에서 리소스를 운영할 수 있습니다. 이를 통해 구축되는 API 서버는 이 격리된 환경 내에서 안전하게 운영됩니다.

2. 세분화된 접근 제어

VPC를 통해 IP 주소 범위, 서브넷, 라우팅 테이블, 네트워크 게이트웨이 등을 완전히 제어할 수 있습니다. 이는 특정 서버만 API에 접근할 수 있도록 하는 용도로 사용 가능합니다.

3. 보안 강화

VPC 내에서 보안 그룹 및 네트워크 ACL을 구성하여 인바운드 및 아웃바운드 트래픽을 세밀하게 제어할 수 있습니다. 이는 허용된 서버에서만 통신이 가능하도록 통제가 필요할 경우 필수적입니다.

서브넷 구성이 필요한 주요 이유

1. 네트워크 세분화

서브넷을 통해 VPC를 더 작은 네트워크 세그먼트로 분할하여 리소스를 논리적으로 그룹화할 수 있습니다. 이를 통해 보안 정책을 더 효과적으로 적용할 수 있습니다.

2. 가용성 향상

여러 가용 영역에 서브넷을 분산 배치함으로써 단일 장애점을 방지하고 시스템의 가용성을 높일 수 있습니다. 이는 안정적인 API 서비스 제공에 중요합니다.

3. 퍼블릭/프라이빗 구분

프라이빗 서브넷에 API 서버를 배치하여 인터넷에서의 직접 접근을 차단할 수 있습니다. 이는 server-to-server 통신만 허용할 경우 필수적인 설정입니다.

4. 트래픽 라우팅 제어

서브넷 구성을 통해 네트워크 트래픽의 흐름을 제어할 수 있습니다. 예를 들어, 특정 서브넷은 인터넷 게이트웨이로 라우팅되지 않도록 설정할 수 있습니다.

적절한 구성

API 서버를 프라이빗 서브넷에 배치하고, 보안 그룹을 통해 허용된 서버 IP에서만 접근할 수 있도록 설정하는 것이 적합합니다. 이렇게 함으로써 인터넷에서의 직접적인 접근은 차단하면서도 필요한 서버 간 통신은 가능하게 할 수 있습니다.
이러한 네트워크 아키텍처는 보안성과 안정성을 모두 고려한 최적의 구성으로, AWS에서 권장하는 모범 사례에도 부합합니다.

cmux: AI 에이전트 시대에 터미널을 다시 설계하면 어떤 모습일까

@jaehong
2026년 3월 27일 8:38 오후
Tech

터미널은 개발자에게 가장 오래된 인터페이스 중 하나다. 그런데 2025년 들어 개발 현장의 풍경이 급변하고 있다. Claude Code, Codex, Gemini CLI 같은 터미널 기반 AI 코딩 에이전트가 폭발적으로 늘어나면서, 개발자들은 동시에 여러 에이전트를 띄워놓고 병렬로 작업하는 새로운 워크플로를 만들어가고 있다. 문제는 기존 터미널이 이런 사용 패턴을 전혀 고려하지 않았다는 것이다. 탭 20개를 열어놓고 어느 에이전트가 끝났는지, 어느 쪽이 입력을 기다리는지 확인하려면 하나하나 클릭해봐야 한다. cmux는 정확히 이 문제를 풀기 위해 등장했다. Ghostty의 렌더링 엔진 libghostty 위에 네이티브 Swift + AppKit으로 구축한 macOS 전용 터미널로, AI 에이전트 워크플로에 최적화된 UI와 자동화 API를 제공한다.

libghostty 위에 쌓은 새로운 레이어

cmux를 이해하려면 Ghostty와의 관계부터 짚어야 한다. HashiCorp 창업자 Mitchell Hashimoto가 만든 Ghostty는 GPU 가속 터미널로, 렌더링 엔진을 libghostty라는 독립 라이브러리로 분리해두었다. cmux는 Ghostty를 포크한 게 아니라, 웹 브라우저가 WebKit을 렌더링 엔진으로 쓰는 것처럼 libghostty를 터미널 렌더링 라이브러리로 활용한다. 즉 Ghostty의 빠른 GPU 렌더링 성능은 그대로 가져가면서, 그 위에 에이전트 워크플로에 특화된 완전히 다른 UX를 올린 것이다.

Hashimoto 본인도 cmux를 "또 하나의 libghostty 기반 프로젝트, 터미널 기반 AI 에이전트 워크플로를 많이 쓰는 사람을 위한 것"이라고 소개하며 호의적인 반응을 보였다. libghostty가 하나의 플랫폼으로 자리 잡아가고 있다는 신호이기도 하다. 예전에 Electron이 데스크톱 앱 생태계를 만들어낸 것처럼, libghostty가 터미널 앱 생태계의 기반 엔진이 될 가능성이 열리고 있는 셈이다.

핵심은 "에이전트가 나를 부를 수 있는 터미널"

cmux의 가장 차별화된 기능은 알림 시스템이다. 기존 터미널에서는 백그라운드 탭에서 돌아가는 프로세스가 끝나도 알 방법이 없었다. cmux는 패널 주위에 파란색 알림 링을 띄우고, 사이드바에 읽지 않은 배지를 표시하며, macOS 네이티브 데스크톱 알림까지 보내준다.

이게 왜 중요한가? AI 코딩 에이전트의 실제 워크플로를 생각해보면 답이 나온다. Claude Code에 "이 모듈 리팩토링해줘"라고 시키고 다른 탭에서 코드 리뷰를 하다가, 에이전트가 "권한 확인이 필요합니다"라고 물어보는 순간을 놓치면 작업이 멈춰선다. cmux는 표준 터미널 이스케이프 시퀀스(OSC 9/99/777)를 자동으로 감지하고, Claude Code hooks나 CLI 명령(cmux notify)으로 직접 알림을 트리거할 수도 있다. 알림에 커스텀 셸 명령을 연결하는 것도 가능하다. 즉 에이전트가 주의를 필요로 할 때 개발자를 능동적으로 불러올 수 있는 양방향 인터페이스가 된 것이다.

세로 탭과 4단계 계층 구조

cmux는 터미널 세션을 Window, Workspace, Pane, Surface 네 단계로 조직한다. 사이드바의 세로 탭이 워크스페이스 역할을 하는데, 각 탭에 현재 git 브랜치, 작업 디렉토리, 리스닝 포트, 알림 텍스트가 표시된다. 이건 단순한 UI 개선이 아니라 컨텍스트 스위칭 비용을 근본적으로 줄이는 설계다.

실제 사용자 Connor의 리뷰가 이를 잘 보여준다. "WIP 작업마다 세로 탭 하나씩, 안에는 한쪽에 Claude, 다른 쪽에 PR과 리소스 브라우저. 작업 전환하면서 정리가 된다." 프로젝트별로 워크스페이스를 나누고, 각 워크스페이스 안에서 에이전트 패널과 브라우저 패널을 분할 배치하면, 여러 에이전트를 동시에 돌리면서도 어떤 작업이 어디까지 진행됐는지 한눈에 파악할 수 있다.

내장 브라우저: 터미널과 웹의 경계를 지우다

cmux에는 스크립팅 가능한 브라우저가 내장되어 있다. Cmd+Shift+B로 터미널 옆에 브라우저 패널을 분할 배치할 수 있다. 여기까지는 "편의 기능" 수준이다. 진짜 흥미로운 건 이 브라우저가 Playwright 수준의 자동화 API를 제공한다는 점이다.

소켓 API를 통해 URL 네비게이션, DOM 쿼리, 클릭, 폼 입력, JavaScript 실행, 스크린샷 캡처, 접근성 트리 추출까지 가능하다. wait-selector, wait-text, wait-url 같은 대기 명령어로 페이지 로딩 상태를 제어할 수 있고, 쿠키와 로컬 스토리지 조작도 된다. 개발자가 터미널에서 에이전트를 돌리면서 동시에 브라우저 기반 테스트나 모니터링을 자동화할 수 있다는 뜻이다. 터미널과 브라우저가 하나의 통합된 자동화 환경으로 작동하는 것이다.

소켓 API: 프로그래밍 가능한 터미널

cmux의 아키텍처에서 가장 주목할 부분은 Unix 소켓 기반의 자동화 API다. ~/.cmux/socket 경로의 소켓에 JSON 형식으로 명령을 보내면 워크스페이스 생성/전환, 패널 분할, 텍스트 입력, 키 입력 전송까지 프로그래밍 방식으로 제어할 수 있다. 보안을 위해 기본적으로 cmux 내부에서 생성된 프로세스만 소켓에 접근할 수 있고(cmuxOnly 모드), 필요하면 allowAll로 열 수도 있다.

여기에 cmux.json 설정 파일로 커스텀 명령과 워크스페이스 템플릿을 정의할 수 있다. 프로젝트별로 "프론트엔드 서버 + 백엔드 서버 + 에이전트"를 미리 레이아웃으로 정의해두고 한 번에 띄우는 식이다. tmux의 .tmux.conf나 tmuxinator가 하던 역할을 GUI 기반으로 대체하면서, 진입 장벽을 크게 낮춘 셈이다.

tmux와의 결정적 차이

tmux는 30년 넘게 터미널 멀티플렉싱의 표준이었다. cmux가 이름에 "mux"를 넣은 건 우연이 아니다. 하지만 접근 방식이 근본적으로 다르다. tmux는 어떤 터미널 안에서든 실행되는 텍스트 기반 멀티플렉서로, SSH 원격 세션 유지 같은 서버 사이드 사용에 여전히 강점이 있다. 반면 cmux는 네이티브 macOS GUI 앱으로, 프리픽스 키 조합(Ctrl+B 후 명령)이 필요 없고 마우스 클릭과 표준 macOS 단축키로 조작한다.

실제 사용자 afruth의 비교가 인상적이다. "세 개 프로젝트를 각각 여러 worktree로 병렬 작업했는데, lazygit이랑 yazi/nvim이랑 같이 쓰니까 여러 ghostty/iTerm 인스턴스를 쫓아다닐 필요 없이 평소보다 생산성이 올랐다. tmux보다 자연스러운 느낌이다." 서버 관리용 tmux와 로컬 개발용 cmux는 사실 경쟁이 아니라 다른 문제를 푸는 도구다. 다만 Claude Code Teams는 이미 cmux의 tmux shim 호환 레이어를 통해 tmux 대신 cmux 분할 패널을 네이티브로 사용할 수 있도록 지원하고 있다.

왜 지금 이 도구가 나왔는가

cmux의 등장은 개발 도구 생태계의 구조적 변화를 반영한다. 2024~2025년 사이에 터미널 기반 AI 코딩 에이전트가 주류로 올라서면서, 개발자의 터미널 사용 패턴이 근본적으로 바뀌었다. 예전에는 터미널이 "명령어를 치고 결과를 보는 곳"이었다면, 지금은 "여러 AI 에이전트가 동시에 일하는 작업장"이 되었다.

IDE 기반 에이전트(Cursor, Windsurf 등)와 달리, Claude Code나 Codex 같은 터미널 기반 에이전트는 기존 개발 환경을 바꾸지 않고 워크플로에 녹아든다는 장점이 있다. 하지만 그 대가로 "에이전트 관리"라는 새로운 과제가 생겼다. 동시에 3~4개 에이전트를 돌리면서 각각의 상태를 추적하고, 입력이 필요할 때 즉시 대응해야 한다. cmux는 이 빈틈을 정확히 파고든다.

Google DeepMind의 리서치 디렉터 Edward Grefenstette가 "주말 내내 써봤는데 정말 대단하다"고 평가하고, Dagster(데이터 파이프라인 프레임워크) 창시자이자 GraphQL 공동 창시자인 Nick Schrock이 "정확히 찾고 있던 제품"이라고 한 건, 이 도구가 실제 파워 유저들의 미충족 수요를 정확히 건드렸다는 방증이다.

남은 질문들

물론 한계도 있다. macOS 전용이라는 플랫폼 제약은 Linux 서버에서 직접 작업하는 개발자를 배제한다. Electron 없이 네이티브 Swift + AppKit으로 구축한 것은 성능 면에서 올바른 선택이지만, 크로스 플랫폼 확장을 어렵게 만드는 트레이드오프이기도 하다. 또한 무료 오픈소스로 제공하고 있어서 장기적인 지속 가능성에 대한 의문도 남는다.

그럼에도 cmux가 제시하는 방향성은 분명하다. AI 에이전트와 함께 일하는 시대에 터미널은 단순한 텍스트 I/O 창이 아니라, 여러 에이전트의 상태를 시각적으로 관리하고 프로그래밍 방식으로 자동화할 수 있는 "에이전트 오케스트레이션 허브"가 되어야 한다는 것이다. libghostty라는 고성능 렌더링 엔진 위에 알림 시스템, 내장 브라우저, 소켓 API라는 세 가지 레이어를 올려서, 기존 터미널이 풀지 못했던 문제를 풀어낸 점에서 cmux는 주목할 가치가 있다.

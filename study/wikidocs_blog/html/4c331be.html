AI Agent 도입 ITSM 아키텍처 = AIOps + Agent

@easyeng
2026년 3월 21일 4:29 오후

🧠 1. 기존 vs AI Agent 도입 후 구조 변화

🔵 기존 구조 (수동 운영 중심)

사용자 → 티켓 생성 → 운영자 처리 → 해결

👉 문제

사람 중심 처리

반복 작업 많음

대응 속도 느림

🟢 AI Agent 도입 구조 (자동화/지능화)

사용자 / 시스템 이벤트
↓
AI Agent
↓
[분석 → 판단 → 실행]
↓
자동 조치 / 티켓 생성 / 보고

👉 핵심 변화
“사람이 처리 → AI가 먼저 처리, 사람은 예외만 대응”

🧱 2. 추가되는 핵심 아키텍처 구성

🧩 기존 + AI 확장 구조

[User / System Event]
↓
[API Gateway]
↓
[AI Agent Layer] ⭐ 추가
↓
[ITSM (iTop/GLPI)]
↓
[Kubernetes / Infra]

🔑 AI Agent Layer 구성

1) LLM (두뇌)

GPT / Claude / 사내 LLM

역할: 판단, 요약, 의사결정

2) Tool (실행 도구)

Kubernetes API

DB 조회

모니터링 시스템

ITSM API

👉 Agent는 “툴을 호출해서 실제 작업 수행”

3) Memory (기억)

과거 장애

해결 패턴

고객 환경 정보

👉 점점 “경험 많은 운영자”처럼 진화

4) Orchestrator (제어)

여러 Agent를 관리

워크플로우 제어

🤖 3. AI Agent 역할 (실무 적용)

🎯 1) L1 Helpdesk 자동화

사용자: "VPN 안됨"
↓
AI Agent:
- 로그 분석
- 원인 파악
- 해결방법 제시 or 자동 조치

👉 티켓 50~70% 감소

🎯 2) 장애 자동 대응 (AIOps)

CPU 90% 발생
↓
AI Agent 판단:
- 트래픽 증가 원인 분석
- Pod 자동 스케일링

🎯 3) Change 자동화

배포 요청
↓
AI Agent:
- 영향도 분석
- 리스크 판단
- 승인 or 보류

🎯 4) CMDB 자동 업데이트

클라우드 / K8s 상태 자동 반영

구성정보 최신화

⚙️ 4. Kubernetes + AI Agent 통합 구조

☸️ 통합 구조

🔑 핵심 연결 포인트

영역 연결 방식

Kubernetes API / Operator
모니터링 Prometheus
로그 ELK
ITSM REST API

💡 핵심 기능

Auto Healing (자동 복구)

Auto Scaling (자동 확장)

Root Cause 분석

💰 5. MSP 관점 과금 모델 변화

🔄 기존 과금
* 사용자 기반
* 티켓 기반

🚀 AI 포함 과금 (고부가가치)

① AI Agent 사용량

호출 횟수 (per request)

토큰 기반 과금

② 자동화 절감 비용 기반

“절감된 인건비의 %”

③ 프리미엄 기능

자동 장애 대응

예측 분석

🎯 추천 구조

기본 ITSM 요금
+ AI Agent 요금 (사용량 기반)
+ 자동화 옵션 (프리미엄)

🚨 6. 반드시 고려해야 할 리스크

⚠️ 1) 잘못된 자동화

AI 오판 → 장애 확대

👉 해결

승인 단계 (Human-in-the-loop)

⚠️ 2) 보안

AI가 인프라 접근

👉 해결

RBAC / 최소 권한

⚠️ 3) 비용 폭증

LLM 호출 비용

👉 해결

캐싱 / 경량 모델 혼용

🎯 최종 아키텍처 (추천)

[Multi-tenant ITSM SaaS]
+
[AI Agent Layer]
+
[Kubernetes Automation]
+
[Monitoring / Logging]

💡 한 줄 핵심 결론

👉 “Docker/K8s가 ‘실행 자동화’라면, AI Agent는 ‘판단 자동화’다”

solidjs의 세분화된 반응성 프로그래밍 모델을 설명하려고 노력함

@sftblw
2024년 11월 4일 10:57 오전

SolidJS가 내세우는 fine-grained reactivity (세분화된 반응성 프로그래밍) 에 대해 얘기할 겁니다. 이거 진짜 알고 나면 별 거 없거든요.

돌고 돌아 SolidJS

저는 좀 이상한 학습 경로를 밟았습니다. Rust WebAssembly 프론트 프레임워크를 쓰고 싶어서 돌고 돌아 SolidJS를 배웠는데, 이유는 Rust GUI 프레임워크인 Sycamore와 Leptos를 배우기 위함이었습니다. 이 둘은 SolidJS에서 모델을 따왔다고 명확히 명시하고 있거든요.

언어 프레임워크 플랫폼 특징
JS / TS SolidJS 웹프론트 이 모델을 유명하게 만듦
Rust sycamore 웹프론트 Leptos보다 먼저 나왔지만 약간 개발속도 부진, 빌더 스타일
Rust leptos 웹프론트 나올 때부터 SSR까지 겸비했고, JSX같은 매크로 지원
Rust floem 데스크톱 Lapce 에디터 개발진의 차기 GUI 프레임워크 (개발 극초기)

이 모델을 배워둬서 손해볼 거 없다는 거죠. 실제로 leptos와 floem 입문에 득을 봤습니다.

저는 오히려 리액트를 잘 모르는데, 이 모델이 전해듣기로는 React Hook 과 비슷하다고 합니다. 차이점은 의존하는 훅이 자동으로 추적된다는 점이라고 하네요.

(* p.s.: 최근 React Hook을 따온 Rust 웹프론트 프레임워크인 Dioxus도 0.5 에서 비슷한 모델로 선회했다고 알고 있는데 이 쪽은 VDOM이 있습니다.)

세분화된 반응성은 엑셀 수식이다.

리액티브 소리만 들으면 머리가 아파옵니다... 으윽.. ReactiveX... 값이 하나만 나오기도 하고 여럿 나오기도 하고... 머리야... 아닌...가?

다행히도 SolidJS가 내세우는 세분화된 반응성은 엑셀 수식처럼 동작합니다. 훨씬 단순하죠.

엑셀을 보면,

  • 데이터의 출처가 되는 셀이 있습니다.
  • 셀의 값을 읽어와서 계산하는 수식이 있습니다. 수식의 값은 수식이 있는 셀에 계산되어 보여집니다.
  • 데이터의 출처가 되는 셀의 값이 바뀌면, 이 셀을 참조하는 다른 수식들도 자동으로 계산되고, 자동으로 값이 바뀝니다.

SolidJS의 프로그래밍 모델도 이렇습니다.

대애충 코드로 봅시다.

// 시그널: 데이터의 출처가 되는 입력 값입니다. get, set 이 가능합니다.
let [getMyVal, setMyVal] = create_signal(123);

// 유도된 시그널: 시그널을 읽어서 계산해서 반환합니다.
let derived_signal = () => {  return getMyVal() * 2; };

// 메모: 유도된 시그널인데, getMyVal() 이 바뀔 때만 실행됩니다.
let memo_value = create_memo( () => { return getMyVal() * 2; };

// 이펙트: 뭔가 다른 짓을 합니다. 사이드이펙트를 낳습니다.
let [getYourVal, setYourVal] = create_signal("값이 2가 아니에요");
create_effect( () => {
    let myVal = getMyVal();
    if (myVal == 2) {
      console.log( "값이 2라뇨!");
      setYourVal("값이 2에요");
    } else {
      setYourVal("값이 2가 아니에요");
    }
});

비동기로 들어가면 create_resource, create_action 같은 것들이 있지만 핵심을 이해하는 데에는 중요하지 않으므로 언급하지 않겠습니다.

자동 추적 스코프가 중요함

잠깐만... 위의 코드에서 유도된 시그널좀 다시 볼게요.

let derived_signal = () => {  return getMyVal() * 2; };

이것만으로 어떻게 getMyVal의 값이 바뀌었을 때 derived_signal 이 자동으로 바뀌죠? 이건 그냥 함수에 불과한데, 이 함수가 getMyVal 을 읽어온다는 걸 누가 어디에서 관리하죠?

그게 바로 추적 스코프 (tracking scope) 입니다. 위 함수는 추적되는 스코프 안에서 실행되어야 자동으로 값이 바뀝니다. create_effect를 다시 볼까요?

create_effect( () => {
  if (getMyVal() == 2) { console.log("2에요!"); }
});

create_effect 는 자동으로 참조하는 값을 추적해주는 추적 스코프를 만들어줍니다. create_effect 안에 넣은 람다 함수는 엑셀 수식 같은 것으로, 그 안에서 참조한 시그널의 값이 바뀌면 자동으로 이 함수를 다시 실행해줍니다.

예, create_effect 처럼 추적되는 범위 안에 있어야 자동으로 값이 바뀐다는 말입니다.

JSX는 자동 추적 스코프임

그러면 위의 derived_signal 은 저렇게 쓰면 추적이 안 된다는 건가요?

let derived_signal = () => {  return getMyVal() * 2; };

create_effect 나 create_memo 안에 넣어야 하나요?

let derived_signal = create_memo( () => {  return getMyVal() * 2; }; );

아... 그게... 이렇게 해도 되긴 하는데, 그럴거면 왜 그냥 람다함수를 유도된 함수라고 소개했을까요?

function MyComponent(): JSX.Element {
  let [getMyVal, setMyVal] = create_signal(123);

  let derived_signal = () => {  return getMyVal() * 2; };

  return (
    <div> {derived_signal()}</div>
  )
}

JSX에서 저런 람다함수를 참조해서 실행하면, SolidJS가 JSX를 컴파일할 때 알아서 저기에 스코프... 가령, create_memo라던가? 를 만들어주기 때문에 문제가 없습니다. (정확히 뭐였는지는 기억 안 남) JSX 안에서 함수를 참조하면 추적 스코프가 자동으로 만들어져서 결과적으로 엑셀 수식처럼 자동으로 계산되는 값이 탄생하게 됩니다.

위 코드를 흔히 보는 예시로 바꿔볼까요?

function MyComponent(): JSX.Element {
  let [getMyVal, setMyVal] = create_signal(0);

  let derived_signal = () => {  return getMyVal() * 2; };

  return (
    <div>
      <button onClick={ setMyVal( getMyVal()+1 ) }>눌러</button>
      <div>{ getMyVal() } 번 누르셨네요. 그 두 배는 { derived_signal() } 이구요</div>
    </div>
  )
}

스코프 말고는 사고방식의 전환이 필요

추적 스코프 개념이 암묵적이어서 그렇지, 그 부분만 주의하면 이 모델은 데이터 흐름이 굉장히 명시적이고, 덤으로 속도까지 빠릅니다.

다만, 데이터 흐름이 단방향으로 강제되는 만큼, 버튼을 눌렀을 때 무언가를 계산해서 내가 바꾼다! 라는 절차지향적 사고가 아니라 소스가 되는 값을 바꾸면 자동으로 값이 계산된다... 라는 데이터 흐름적 사고로 머릿속의 프로그래밍 모델을 바꿀 필요가 있습니다.

예를 들자면, bool 시그널이 false이고, true로 바뀌면 effect를 실행한 뒤 다시 false로 바꾸는 거 같은 방식으로 트리거하는 함수를 짤 수도 있습니다. 예쁘진 않은데...


let [getDoAction, setDoAction] = create_signal(false);

create_effect( () => {
  if (getDoAction() == true) {
    console.log("DoAction 했습니다!");
    setDoAction(false);
  }
});

이런 것도 자꾸 짜다보면, 또 연습하고 문서 보고 공부하다보면 더 나은 방법이 보일테니까요.

무엇보다도 좋은 점은 값이 알아서 바뀐다는 점입니다. 이벤트 핸들러와 씨름할 필요가 없다는 게 제일 행복한 포인트가 아닌가 생각해봅니다.

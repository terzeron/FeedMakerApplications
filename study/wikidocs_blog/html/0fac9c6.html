🧩 [3회] SoC 내부를 해부해보자

— CPU, GPU, NPU, 그리고 IP 블록의 역할

스마트폰으로 사진을 찍을 때, 어떤 일이 벌어질까요?
셔터를 누르는 순간, 카메라 센서가 빛을 받아들이고, 이미지 신호 처리기가 색을 다듬고, NPU가 피사체를 인식해 배경을 흐리게 만들고, GPU가 화면에 미리보기를 띄웁니다.
이 모든 일이 1초도 안 되는 사이에, 하나의 칩 안에서 동시에 일어납니다.

그 칩이 바로 SoC입니다.
2회차에서 SoC가 "하나의 칩에 시스템 전체를 넣은 것"이라고 했는데, 그럼 그 안에는 정확히 무엇이 들어 있을까요?
오늘은 SoC의 내부 지도를 함께 펼쳐봅니다.

SoC는 도시다 — 블록 구조로 이해하기

SoC 내부를 이해하는 가장 좋은 비유는 도시입니다.
도시에는 시청(CPU), 공장(GPU), 연구소(NPU), 도로망(버스), 창고(메모리) 등 각자 역할이 다른 건물들이 있고, 이들이 서로 연결되어 하나의 시스템을 이룹니다.

SoC도 마찬가지입니다.
각각의 기능 단위를 IP 블록(Intellectual Property Block) 이라고 부릅니다.
이름이 조금 낯설지만, 쉽게 말하면 "검증된 회로 부품"입니다.
레고처럼 이미 만들어진 블록을 가져다 조합하는 방식으로 SoC를 설계합니다.
ARM의 CPU 코어, 이미지 처리기, USB 컨트롤러 등이 모두 IP 블록입니다.

아래에 SoC의 한 예를 보여 줍니다.

그렇다면 SoC 안에 있는 주요 블록들을 하나씩 살펴봅시다.

CPU — 총책임자, 하지만 만능은 아니다

CPU(Central Processing Unit) 는 흔히 "두뇌"라고 불립니다.
맞는 말이지만, 더 정확하게는 복잡한 판단을 내리는 총책임자입니다.
"이 앱을 실행해라", "이 파일을 열어라", "저 블록에 이 데이터를 보내라" 같은 순서 있는 명령들을 처리합니다.

CPU는 매우 유연합니다.
어떤 연산이든 처리할 수 있습니다.
하지만 그만큼 회로가 복잡하고, 한 번에 많은 작업을 병렬로 처리하기에는 적합하지 않습니다.
스마트폰용 SoC에는 보통 여러 개의 코어가 들어가는데, 고성능 코어(빅 코어)와 저전력 코어(리틀 코어)를 섞어 배치합니다.
게임을 할 때는 빅 코어가 활성화되고, 유튜브를 백그라운드에서 틀어두는 정도면 리틀 코어가 처리하는 방식입니다.
이를 빅-리틀(big.LITTLE) 구조라고 합니다.

GPU — 단순한 일을 엄청 빠르게, 동시에

GPU(Graphics Processing Unit) 는 원래 화면에 그림을 그리기 위해 만들어졌습니다.
화면 하나를 채우려면 수백만 개의 픽셀을 각각 계산해야 하는데, CPU처럼 하나씩 처리하다간 화면이 뚝뚝 끊깁니다.

GPU는 이 문제를 병렬 처리로 해결합니다.
CPU가 뛰어난 직원 몇 명이라면, GPU는 단순 작업을 동시에 처리하는 수천 명의 직원 집단입니다.
요즘은 AI 연산도 결국 대량의 행렬 곱셈이라 GPU가 딥러닝에도 많이 쓰입니다.
엔비디아가 AI 붐과 함께 주목받는 이유도 여기에 있습니다.

NPU — AI 전용 가속기, 왜 따로 만들까?

요즘 스마트폰 광고에 빠지지 않는 단어가 있습니다.
"온디바이스 AI", "AI 카메라". 이것들의 핵심이 NPU(Neural Processing Unit) 입니다.

그런데 의문이 생깁니다.
GPU도 AI 연산을 잘 한다고 했는데, 왜 NPU를 따로 만들까요?

답은 전력 효율에 있습니다.
GPU는 빠르지만 전력을 많이 씁니다.
스마트폰에서 GPU로 AI 연산을 계속 돌리면 배터리가 순식간에 닳아버립니다.
NPU는 AI에서 가장 많이 쓰이는 연산(행렬 곱셈, 활성화 함수 등)에 최적화된 회로만 담아, 같은 연산을 GPU보다 훨씬 적은 전력으로 처리합니다.
목적을 좁힌 만큼 효율이 극대화되는 것입니다.

애플의 Neural Engine, 퀄컴의 Hexagon NPU, 삼성의 MX Core가 모두 이에 해당합니다.

그 외 주요 IP 블록들

SoC 안에는 CPU, GPU, NPU 말고도 수십 개의 IP 블록이 더 있습니다.
대표적인 것들을 간략히 살펴봅니다.

ISP(Image Signal Processor) 는 카메라에서 들어오는 원시 이미지 데이터를 사람이 보기 좋은 사진으로 변환합니다.
노이즈 제거, 색 보정, HDR 합성이 모두 여기서 이루어집니다.

DSP(Digital Signal Processor) 는 오디오나 통신 신호 같은 연속적인 데이터를 전문으로 처리합니다.
통화 중 소음을 제거하거나, 음악 재생 시 이퀄라이저를 적용하는 것이 DSP의 역할입니다.

메모리 컨트롤러는 SoC와 외부 DRAM 사이의 교통을 정리합니다. 데이터를 언제, 어떤 순서로 읽고 쓸지를 제어합니다.

모뎀(Modem) 은 5G, LTE 같은 통신 신호를 처리합니다.
퀄컴 스냅드래곤 칩에는 모뎀이 SoC에 통합되어 있고, 애플은 한동안 별도 칩을 써왔다가 최근 자체 모뎀 개발에 성공했습니다.

IP 블록은 어떻게 연결되는가?

이렇게 다양한 블록들이 서로 데이터를 주고받으려면 공통 연결망이 필요합니다.
이것이 바로 버스(Bus) 또는 인터커넥트(Interconnect) 입니다.
도시로 치면 도로망입니다.

SoC에서는 ARM이 만든 AXI(Advanced eXtensible Interface) 프로토콜이 사실상의 표준으로 쓰입니다.
CPU가 "메모리에서 이 주소의 데이터를 가져와라"라고 명령을 내리면, AXI 버스를 통해 메모리 컨트롤러까지 요청이 전달되고, 데이터가 돌아오는 방식입니다.
AXI 버스 프로토콜은 이후 회차에서 별도로 자세히 다룰 예정입니다.

설계자 입장에서 보면

이 모든 IP 블록은 누군가가 RTL 코드로 설계하고, 또 누군가가 검증한 결과물입니다.
대형 칩 회사에서는 수십 명의 팀이 각 블록을 맡아 설계하고, 전체 SoC가 제대로 동작하는지 통합 검증을 별도로 수행합니다.

흥미로운 점은, 모든 블록을 직접 만들지 않아도 된다는 것입니다.
ARM의 CPU 코어처럼 이미 검증된 IP를 라이선스 비용을 내고 구매해 사용하는 경우가 많습니다.
팹리스 회사들이 빠른 속도로 새 칩을 출시할 수 있는 것도 이 IP 생태계 덕분입니다.

다음 편 예고

SoC의 내부 구조를 알았으니, 이제 자연스럽게 이런 질문이 떠오릅니다.
"그런데 설계를 다 하고 나서, 이게 정말 제대로 동작한다는 걸 어떻게 확인하죠?" 다음 회차에서는 설계와 검증, 왜 둘 다 필요한가를 주제로, 버그 하나가 얼마나 큰 비용을 초래하는지 그리고 현업에서 설계 대비 검증 인력이 왜 더 많은지를 다룹니다.

💬 **한 줄 정리 : SoC 안에는 CPU(판단), GPU(병렬 처리), NPU(AI 가속) 등 목적이 다른 IP 블록들이 버스로 연결되어 하나의 시스템을 이룬다. **

30. 동시성 프로그래밍

동시성 프로그래밍은 여러 작업이 동시에 실행되는 프로그램을 작성하는 기법을 의미한다. 순차(sequential) 실행은 하나의 프로세스가 끝나야 다음 프로세스를 실행하고, 한 번에 한 가지 일만 할 수 있다. 동시(concurrent) 실행은 두 프로세스가 하나의 CPU코어를 공유하고 작업 시간을 분할하여 수행하는 구조이다. 하나의 코어를 공유하여 사용하기 때문에 물리적으로는 한 번에 한 가지 작업만 가능하다. 두 프로세스의 전체 수행 시간은 순차 실행과 동일하다. 병렬(parallel) 실행은 두 프로세스가 각각 독립적으로 코어를 할당 받아 동시에 병렬로 작업을 수행한다.

Rust의 동시성 모델은 스레드 기반 동시성과 비동기 프로그래밍 두 가지로 나뉜다. Rust의 메모리 안정성은 동시성 원칙에서 최우선 되는 속성으로 복잡한 동시성 문제를 안전하게 처리할 수 있도록 도와준다. 이 동시성을 보장하기 위해 러스트가 갖고 있는 몇 가지 규칙은 다음과 같다.

  • 데이터 경쟁(Data Race) 방지

  • 소유권(Ownership) 시스템

  • 소유권 대여(Borrowing)와 수명주기(Lifetimes)

  • 메시지 전달(Message Passing) 방식

  • 동시성을 다루는 다양한 추상화(Mutex, RwLock, Condvar 등) 기능

특히 소유권(Ownership) 원칙은 변수의 공유를 막아 여러 스레드가 동시에 자료에 접근할 수 없도록 강제한다. 스레드 안전성을 컴파일 타임에 보장하여, 교착 상태나 경쟁 상태와 같은 동시성 문제들의 발생 가능성이 다른 언어에 비해 현저히 낮다.

또한, async/await와 같이 현대 프로그래밍 언어가 제공하는 동시성 기능들을 지원하고 있으며, 이를 통해 동시성을 쉽고 간편하고 안전하게 제어할 수 있다.

동시성 요소(concurrency primitives)는 여러 스레드나 프로세스가 동시에 작업을 수행할 수 있도록 하는 기본적인 도구이다. 이러한 요소는 동시성 문제를 해결하고 데이터 무결성을 유지하기 위해 사용된다. 주요 동시성 요소는 다음과 같다.

  • 뮤텍스(Mutex): 상호 배제를 위한 도구로, 여러 스레드가 동시에 특정 코드 블록에 접근하지 못하도록 한다. 한 스레드가 뮤텍스를 잠그면 다른 스레드는 잠금을 해제할 때까지 대기해야 한다.

  • 세마포어(Semaphore): 제한된 수의 스레드가 공유 자원에 접근할 수 있도록 제어한다. 카운팅 세마포어와 바이너리 세마포어가 있으며, 카운팅 세마포어는 여러 스레드가 동시에 접근할 수 있는 수를 지정한다.

  • 모니터(Monitor): 뮤텍스와 조건 변수(Condvar)를 결합하여 객체의 상태를 보호하고 동기화를 쉽게 한다. 모니터 내부의 코드 블록에 대한 접근은 뮤텍스를 통해 제어되며, 조건 변수는 특정 조건이 충족될 때까지 스레드를 대기 상태로 만든다.

  • 조건 변수(Condition Variable): 스레드가 특정 조건이 충족될 때까지 대기하도록 하는 데 사용된다. 조건 변수를 사용할 때는 뮤텍스와 함께 사용하여 데이터의 일관성을 보장한다.

  • 읽기-쓰기 잠금(Read-Write Lock): 여러 스레드가 읽기 작업을 동시에 수행할 수 있도록 하지만, 쓰기 작업이 필요할 경우에는 독점적으로 잠금을 허용한다.

  • 이벤트(Event): 스레드가 특정 사건(예: 데이터 수신)을 기다릴 수 있도록 하는 신호 메커니즘이다. 하나의 스레드가 이벤트를 설정하면 대기 중인 다른 스레드가 실행을 재개할 수 있다.

두 스레드가 데이터를 공유하는 상황에서 두 스레드 모두가 나머지 하나보다 더 오래 존재한다는 사실이 보장되지 않는다면, 어떤 스레드도 데이터의 소유권을 가질 수 없다. 공유되는 데이터는 두 스레드보다 더 오래 존재해야만 한다.

static 변수는 프로그램 자체가 소유권을 가지기 때문에 모든 스레드보다도 오래 존재할 수 있다. 다음 코드에서 두 스레드 모두 X에 접근할 수 있지만, X를 소유할 수는 없다.

static X: [i32; 3] = [1, 2, 3];

thread::spawn(|| dbg!(&X));
thread::spawn(|| dbg!(&X));

static으로 선언된 변수는 상수 값을 가지고 프로그램 시작전에 생성되어 어떤 스레드도 static변수로부터 값을 빌려올 수 있다.

또 다른 데이터 공유 방법은 값의 할당을 유출시키는 방법이다. Box<T>는 소유권을 가지고 있는 힙 메모리에 값을 할당하며, Box의 범위가 끝나면 자동으로 해제된다. 하지만 Box::leak를 사용하면, 이 메모리를 명시적으로 해제하지 않고, 해당 값을 static 수명주기를 가지는 참조로 변환할 수 있다.

이때부터Box는 프로그램이 종료될 때까지 존재하게 되고 어느 스레드에서도 값을 빌려 갈 수 있게 된다.

let x: &'static [i32; 3] = Box::leak(Box::new([1, 2, 3]));

thread::spawn(move || dbg!(x));
thread::spawn(move || dbg!(x))

move 클로저가 값의 소유권을 가져가는 것처럼 보이지만, x는 단순히Box의 참조이다. 'static으로 선언되었다고 해서 이 값이 프로그램 시작 전에 만들어진다는 것은 아니다. 이 값은 프로그램이 종료될 때까지 유지된다.

Box를 유출시키게 되면, 메모리에 어떤 객체를 할당했지만 객체를 삭제하고 메모리 할당을 해제하지 않는 것과 같다. 이러한 패턴이 몇 번 존재하지 않고 프로그래머가 명확히 이를 관리할 수 있으면 상관이 없지만, 이러한 패턴이 반복되면 프로그램의 메모리가 점차 부족해질 것이다.

스레드 사이에서 공유된 데이터가 확실히 삭제되고 메모리에서 할당 해제되게 하려면, 해당 데이터의 소유권을 포기해서는 안된다. 소유권을 공유하여 해당 데이터의 소유자들을 지속적으로 관리함으로써 더 이상 해당 데이터의 소유자가 없을 때 객체를 삭제할 수 있다.

Box::leak을 사용하지 않고도 여러 스레드에서 안전하게 데이터를 공유하면서도 필요할 때 자동으로 해제되도록 하려면 소유권을 공유해야 한다. 이를 위해 Arc<T>와 Mutex<T> 혹은 RwLock<T> 같은 동기화 도구를 사용할 수 있다.

30.1 스레드 만들기

표준 라이브러리는 POSIX Threads(pthreads) 및 Windows Threads 위에 안전한 스레드 API를 제공한다. std::thread 모듈을 활용하여 여러 스레드를 생성하고, 데이터를 공유하며, 동기화할 수 있다.

thread::spawn은 독립적으로 실행되는 새로운 스레드를 생성할 때 사용하는 함수이다. thread::spawn을 호출하면, JoinHandle이 반환되며 이를 통해 스레드의 결과를 처리하거나 종료를 기다릴 수 있다.

[EX30-1-1]

use std::thread;

fn main() {
    let handle = thread::spawn(|| {
        for i in 1..10 {
            println!("새로운 스레드: {}", i);
            std::thread::sleep(std::time::Duration::from_millis(100));
        }
    });

    for i in 1..5 {
        println!("메인 스레드: {}", i);
        std::thread::sleep(std::time::Duration::from_millis(100));
    }

    handle.join().unwrap(); // 새 스레드가 종료될 때까지 대기
}

  1. thread::spawn으로 생성된 스레드는 비동기적으로 실행된다. 생성된 스레드가 완료되기 전에 메인 스레드가 종료되면, 생성된 스레드도 종료된다.

  2. thread::spawn은 JoinHandle을 반환하며, join 메서드로 해당 스레드의 종료를 기다릴 수 있다. 스레드에서 반환된 값은 JoinHandle을 통해 얻을 수 있다.

  3. thread::spawn에 전달하는 클로저는 반드시 'static 수명을 가져야 한다.

아래의 예제는 클로저가 메인 스레드의 변수를 캡쳐하여 클로저의 환경으로 이동한다. 스레드에서 사용하는 변수는 클로저 내에서 move로 소유권을 가져와야 할 수 있다.

[EX30-1-2]

use std::thread;

fn main() {
    let handle=thread::spawn(move|| {
    let data=vec![1, 2, 3];
        println!("Data: {:?}", data);
    });

    handle.join().unwrap();
}

아래의 예제는 새 스레드를 생성하고 1부터 10의 합을 계산하는 클로저를 실행한다. 파라미터들은 thread::spawn 함수의 파라미터로 보내져서 join 메서드를 통해 반환 값을 받을 수 있다.

[EX30-1-3]

use std::thread;

fn main() {
    let handle = thread::spawn(|| {
        let sum: u32 = (1..=10).sum();
        sum
    });

    let result = handle.join().unwrap();
    println!("스레드에서 반환된 값: {}", result);
}

메인 스레드의 프로그램이 완료되면 생성된 모든 스레드가 스레드의 작업이 끝나지 않아도 모두 실행을 종료하게 된다. 아래 코드의 실행 결과는 매번 다를 수 있다.

[EX30-1-4]

use std::thread;
use std::time::Duration;

fn main() {
    thread::spawn(|| {
    for i in 1..10 {
        println!("hi number {} from the spawned thread!", i);
        thread::sleep(Duration::from_millis(1));
    }
});

for i in 1..5 {
        println!("hi number {} from the main thread!", i);
        thread::sleep(Duration::from_millis(1));
    }
}

thread::sleep 함수는 짧은 시간 동안 실행을 멈추어 다른 스레드에서 코드를 실행시킬 수 있게 한다. 운영체제마다 스케쥴링 방법이 다르기 때문에 순서와 결과를 보장할 수 없다.

thread::spawn을 사용할 때 클로저 대신 별도의 함수로 스레드 작업을 정의할 수도 있다. 클로저가 아닌 함수를 사용하려면 함수 시그니처와 매개변수의 소유권을 적절히 처리해야 한다.

아래는 thread::spawn에서 클로저를 사용하지 않고 함수를 호출하는 예제이다.

[EX30-1-5]

use std::thread;

fn thread_function(start: u32, end: u32) {
    for i in start..end {
        println!("스레드: {}", i);
        std::thread::sleep(std::time::Duration::from_millis(100));
    }
}

fn main() {
    // thread::spawn 호출 시, 함수 포인터를 전달
    let handle = thread::spawn(|| thread_function(1, 10));

    // 메인 스레드 작업
    for i in 1..5 {
        println!("메인 스레드: {}", i);
        std::thread::sleep(std::time::Duration::from_millis(100));
    }

    // 스레드 종료 대기
    handle.join().unwrap();
}

위 코드는 함수 thread_function을 호출하는 클로저를 전달하지만, 클로저 자체를 제거할 수도 있다.

thread::spawn은 함수 포인터를 받을 수 있기 때문에 아래와 같은 코드도 가능하다

[EX30-1-6]

use std::thread;

// 스레드에서 실행할 함수
fn thread_function() {
    for i in 1..10 {
        println!("스레드: {}", i);
        std::thread::sleep(std::time::Duration::from_millis(100));
    }
}

fn main() {
    // thread::spawn에 함수 포인터 전달
    let handle = thread::spawn(thread_function);

    // 메인 스레드 작업
    for i in 1..5 {
        println!("메인 스레드: {}", i);
        std::thread::sleep(std::time::Duration::from_millis(100));
    }

    // 스레드 종료 대기
    handle.join().unwrap();
}

thread::spawn은 FnOnce 트레잇을 구현하는 함수 포인터나 클로저를 인수로 받을 수 있다.

클로저를 사용하지 않으려면 함수 포인터를 전달하면 된다. 함수에 매개변수를 전달해야 한다면, 별도의 클로저를 사용하거나 매개변수를 캡처하는 방법을 고려해야 한다.

[EX30-1-7]

use std::thread;

// 스레드에서 실행할 함수
fn thread_function(start: u32, end: u32) {
    for i in start..end {
        println!("스레드에서: {}", i);
        thread::sleep(std::time::Duration::from_millis(100));
    }
}

fn main() {
    // 클로저를 사용하여 매개변수 전달
    let handle = thread::spawn(|| thread_function(1, 5));

    for i in 1..=5 {
        println!("메인 스레드에서: {}", i);
        thread::sleep(std::time::Duration::from_millis(100));
    }

    handle.join().unwrap();
}

함수 포인터와 매개변수의 조합을 지원하지 않으므로 매개변수를 전달하려면 move 클로저를 사용하여 함수 호출을 감싸야 한다.

[EX30-1-8]

use std::thread;

// 스레드에서 실행할 함수
fn thread_function(start: u32, end: u32) {
    for i in start..end {
        println!("스레드에서: {}", i);
        thread::sleep(std::time::Duration::from_millis(100));
    }
}

fn main() {
    let start = 1;
    let end = 5;

    // 클로저 없이 매개변수를 직접 전달하려면 move 클로저로 감싸야 함
    let handle = thread::spawn(move || thread_function(start, end));

    for i in 1..=5 {
        println!("메인 스레드에서: {}", i);
        thread::sleep(std::time::Duration::from_millis(100));
    }
    handle.join().unwrap();
}

30.3 move 클로저

move 클로저는 클로저가 사용하는 외부 변수를 캡처할 때, 해당 변수의 소유권을 강제로 이동시키는 클로저이다. 이는 멀티스레딩 상황에서 유용하며, 소유권 시스템과 호환되는 안전한 방식으로 데이터를 다룰 수 있도록 한다.

클로저가 외부 변수를 사용할 때 기본적으로 참조를 캡처한다. 하지만 move를 사용하면 해당 변수의 소유권이 클로저로 이동한다. 클로저는 더 이상 외부 변수에 접근할 수 없게 되고, 해당 변수는 클로저 내에서만 사용된다.

스레드 모델에서 데이터의 소유권이 명확히 정의되지 않으면 데이터 경합이 발생할 수 있다. move를 사용하여 데이터의 소유권을 클로저로 이전하면, 스레드 간 데이터 경합을 방지할 수 있다.

move 클로저(move || { ... })는 소유권을 이전(move)하여 캡처하는 클로저이다. 하지만 Copy 트레잇을 구현한 값은 복사할 수 있기 때문에, 클로저 내부로 이동하는 것이 아니라 복사된다.

[EX30-3-1]

fn main() {
    let x = 10; // i32는 Copy
    let closure = move || {
        println!("{}", x); // x가 클로저 내부로 이동하지만, Copy라서 복사됨
    };

    closure();

    println!("{}", x); // 여전히 x를 사용할 수 있음!
}

move 클로저를 사용하지 않은 경우 클로저가 x의 참조를 캡처하므로, x는 클로저 호출 이후에도 사용할 수 있다.

[EX30-3-2]

fn main() {
    let x = 10;
    let closure = || println!("x: {}", x);

    closure();
   
    println!("x는 여전히 사용할 수 있음: {}", x);

    let message = String::from("Hello from thread!");
    let closure = || println!("{}", message);

    closure();

    println!("message는 여전히 사용할 수 있음: {}", message);
}

move 키워드를 사용했기 때문에 x, y의 소유권이 클로저로 이동한다. 이후 x, y를 사용하려 하면 컴파일 오류가 발생한다.

[EX30-3-3]

fn main() {
    let x = String::from("Hello");
    let y = String::from("Rust");
    let closure = move || {
        println!("x: {}, y: {}", x, y);
    };

    closure();
    // println!("{}", x); // 컴파일 오류: x의 소유권이 이동됨
    // println!("{}", y); // 컴파일 오류: y의 소유권이 이동됨
}

thread::spawn은 클로저에 전달된 데이터의 소유권이 명확히 정의되지 않으면 컴파일 오류를 발생한다. 이때 move 클로저를 사용하여 데이터의 소유권을 스레드로 이동시킬 수 있다.

[EX30-3-4]

use std::thread;

fn main() {
    let message = String::from("Hello from thread!");
    let handle = thread::spawn(move || {
        println!("{}", message);
    });

    handle.join().unwrap();
    // println!("{}", message); // 컴파일 오류: message의 소유권이 스레드로 이동됨
}

message의 소유권이 move 클로저로 이동했기 때문에, 스레드 내부에서 안전하게 사용할 수 있다. message는 이후 메인 스레드에서 사용할 수 없다.

30.4 범위제한 스레드(Scoped Threads)

std::thread::spawn을 통해 생성되는 스레드는 부모 스레드의 범위와 독립적인 수명 주기를 갖게 된다. 이렇게 독립적으로 실행되는 스레드는 move로 소유권을 가져가야 하며, 부모 범위의 변수에 접근하려면 소유권을 이전해야 하는 제약이 있다.

범위 제한 스레드는 스레드의 수명을 명시적으로 제어하여, 스레드가 부모 범위의 데이터를 안전하게 참조할 수 있도록 해준다. 이를 통해 소유권 및 수명 규칙과 충돌하지 않으면서 스레드를 생성하고 사용할 수 있다. 범위 제한 스레드는 주로 std::thread::scope를 사용하여 구현된다.

[EX30-4-1]

use std::thread;

fn foo() {
    let s = String::from("Hello");

    thread::spawn(|| {
        println!("Lenght : {}", s.len());
    });
}

fn main() {
    foo();
}

5 | thread::spawn(|| {
| ^^ may outlive borrowed value `s`
6 | println!("Lenght : {}", s.len());
| - `s` is borrowed here

어떤 스레드가 반드시 특정 범위에서만 존재하는 것이 확실할 경우, 이 스레드는 지역 변수의 소유권을 빌려올 수 있다. 스레드 역시 지역변수와 같이 특정 범위에서만 존재하기 때문이다.

[EX30-4-2]

use std::thread;

fn main() {
    let s = String::from("Hello");

    thread::scope(|scope| {
        scope.spawn(|| {
            println!("Length : {}", s.len());
        });
    });
}

  1. thread::scope 함수가 완료되면 그 안에서 생성된 모든 스레드들이 종료했음이 보장되기 때문에, 그 때 빌렸던 데이터들을 다시 반환할 수 있다.

  2. 일반적인 빌림 규칙이 적용된다: 한 스레드에 의한 가변 빌림 또는 여러 스레드에 대한 불변 빌림 중 하나만 가능하다.

아래 코드는 범위 제한 스레드를 사용하여 부모 범위의 데이터를 수정하는 예제이다.

[EX30-4-3]

use std::thread;

fn main() {
    let mut data = vec![10, 20, 30];

    thread::scope(|s| {
        // 첫 번째 스레드에서만 data를 수정하도록 함
        s.spawn(|| {
            for i in &mut data {
                *i += 1;
            }

            println!("첫 번째 스레드에서 변경된 데이터: {:?}", data);
        });
    });

    println!("메인 스레드에서 최종 데이터: {:?}", data);
}

위의 코드에는 몇 가지 문제가 있다. 가장 큰 문제는 data를 여러 스레드에서 안전하게 공유하지 않고 직접 수정하려고 시도하는 것이다. Rust의 안전성 모델에서는 데이터 경쟁(data race)을 방지하기 위해 명시적인 동기화 메커니즘을 사용해야 한다.

아래 코드는 다중 스레드를 사용하는 예제이다.

std::sync::Mutex와 std::sync::Arc를 사용하여 여러 스레드가 안전하게 데이터를 공유하고 수정할 수 있다.

[EX30-4-4]

use std::sync::{Arc, Mutex}; // 동기화를 위한 Arc, Mutex 가져오기
use std::thread; // 스레드 사용을 위한 모듈

fn main() {
    // Arc + Mutex로 데이터 보호
    let data = Arc::new(Mutex::new(vec![10, 20, 30]));
   
    thread::scope(|s| {
        let data_clone = Arc::clone(&data); //  Arc를 클론하여 소유권 공유

        s.spawn(move || { // 🔹 새로운 스레드 생성
            let mut data = data_clone.lock().unwrap(); //  Mutex 잠금 (lock)
            for i in &mut *data { // 🔹 벡터 내부 요소 변경
                *i += 1;
            }

            println!("첫 번째 스레드에서 변경된 데이터: {:?}", data);
        });
    });

    println!("메인 스레드에서 최종 데이터: {:?}", *data.lock().unwrap());
}

crossbeam::scope는 안전하게 범위 제한 스레드를 생성할 수 있게 해주는 crossbeam 라이브러리의 기능이다. std::thread::spawn은 스레드가 부모 범위의 변수에 접근하기 위해 소유권을 가져가야 하는 제약이 있다. 반면 crossbeam::scope는 부모 범위의 변수에 안전하게 접근할 수 있도록 보장하며, 범위 내에서 스레드가 모두 종료되도록 한다. 따라서 소유권을 이동(move)하지 않아도 부모 범위의 변수를 참조할 수 있어 메모리 안전성을 유지하면서 쉽게 멀티스레딩을 구현할 수 있다.

[dependencies]
crossbeam = "0.8"

[EX30-4-5]

use crossbeam::scope;

fn main() {
    let numbers = vec![1, 2, 3, 4, 5];
    let mut sum = 0;

    scope(|s| {
        // 첫 번째 스레드: 벡터의 첫 번째 절반을 더함
        s.spawn(|_| {
            let partial_sum: i32 = numbers[..3].iter().sum();
            println!("첫 번째 스레드에서 부분 합계: {}", partial_sum);
        });

        // 두 번째 스레드: 벡터의 두 번째 절반을 더함
        s.spawn(|_| {
            let partial_sum: i32 = numbers[3..].iter().sum();
            println!("두 번째 스레드에서 부분 합계: {}", partial_sum);
        });
    }).unwrap(); // 모든 스레드가 종료될 때까지 기다림

    println!("전체 합계: {}", sum);
}

  • crossbeam::scope를 사용할 때는 스레드가 범위를 벗어나기 전에 종료되도록 코드를 작성해야 한다.

  • crossbeam::scope는 블로킹 작업이나 CPU 집중 작업에 적합하며, Tokio와 같은 비동기 프레임워크 내에서는 tokio::task::spawn_blocking을 사용할 것을 권장한다.

30.5 join

join 메서드는 스레드의 실행이 끝날 때까지 기다리고, 스레드에서 반환된 값을 가져오는 데 사용된다. 스레드가 반환한 값을 Result<T, E> 형식으로 반환한다.

[EX30-5-1]

use std::thread;

fn main() {
    let handle = thread::spawn(|| {
        for i in 1..5 {
            println!("새 스레드: {}", i);
            thread::sleep(std::time::Duration::from_millis(500));
        }

        42 // 스레드에서 반환할 값
    });

    // join을 호출하여 스레드가 종료될 때까지 대기
    let result = handle.join().unwrap();

    println!("스레드에서 반환된 값: {}", result);
    println!("메인 스레드 종료");
}

  1. join은 해당 스레드가 종료될 때까지 메인 스레드를 차단(block)한다.

    이를 통해 스레드가 실행을 완료할 때까지 기다릴 수 있다.

  2. join 메서드는 스레드에서 반환된 값을 포함하는 Result<T, Box<dyn Any + Send + 'static>>를 반환한다.

  3. 정상적으로 실행되면 Ok(T)를 반환하며, 패닉이 발생하면 Err를 반환한다.

스레드 내부에서 패닉이 발생하면 join은 Err를 반환하므로, 이를 적절히 처리해야 한다.

예를 들어, unwrap을 호출하면 패닉이 전파된다. join 결과에 대해 unwrap을 호출하면, 스레드에서 패닉이 발생했을 때 unwrap이 다시 패닉을 유발한다. 따라서, 반드시 match나 if let을 사용하여 Err를 처리해야 한다.

[EX30-5-2]

use std::thread;

fn main() {
    let handle = thread::spawn(|| {
        panic!("Something went wrong in the thread!");
    });

    match handle.join() {
        Ok(_) => println!("Thread finished successfully."),
        Err(e) => {
            println!("Thread panicked.");

            if let Some(panic_msg) = e.downcast_ref::<&str>() {
                println!("Panic message: {}", panic_msg);
            } else {
                println!("Panic message could not be decoded.");
            }
        }
    }
}

아래는 다중 스레드에서 join을 사용하는 예제이다.

[EX30-5-3]

use std::thread;

fn main() {
    let handles: Vec<_> = (1..=5)
        .map(|i| {
            thread::spawn(move || {
                println!("스레드 {} 실행", i);
                i * 2 // 스레드에서 반환된 값
            })
        })
        .collect();

    for handle in handles {
        let result = handle.join().unwrap();

        println!("스레드에서 반환된 값: {}", result);
    }

    println!("모든 스레드가 종료되었습니다.");
}

join의 역할:

  • 스레드가 종료될 때까지 기다린다.

  • 스레드에서 반환된 값을 가져올 수 있다.

  • 패닉 발생 시 이를 처리할 수 있다.

언제 사용할까?

  • 스레드의 종료를 보장해야 할 때.

  • 스레드에서 반환된 값을 활용해야 할 때.

  • 스레드 내부의 오류를 처리해야 할 때.

crossbeam::scope를 사용하는 경우에는 명시적으로 join을 호출하지 않아도 된다. 범위가 끝날 때까지 모든 스레드가 종료되도록 보장하므로, 스레드 종료를 명시적으로 기다릴 필요가 없다.

[EX30-5-4]

use crossbeam::scope;

fn main() {
    scope(|s| {
        s.spawn(|_| {
            println!("스레드 1 실행 중");
        });

        s.spawn(|_| {
            println!("스레드 2 실행 중");
        });
    }).unwrap(); // 모든 스레드가 종료될 때까지 기다림

    println!("모든 스레드 종료.");
}

30.6 Send와 Sync

Send와 Sync는 멀티스레드 환경에서의 안전한 데이터 이동 및 공유를 지원하는 중요한 마커 트레잇이다. 소유권 시스템과 함께 동작하여 데이터 경합을 방지하고, 다중 스레드 환경에서 메모리 안전성을 보장한다.

Send 트레잇은 한 스레드에서 다른 스레드로 소유권을 안전하게 이동할 수 있는 타입을 나타낸다.

  • 기본 타입(i32, u64, String, Vec<T>, 등)은 대부분 Send를 자동으로 구현한다.

  • 스레드 안전성을 보장하지 않는 타입(예: 비동기 I/O에서 파일 핸들, Rc<T>)은 Send 트레잇을 구현하지 않는다.

[EX30-6-1]

use std::thread;

fn main() {
    let data = String::from("Hello, world!");
    let handle = thread::spawn(move || {
        println!("{}", data);  // data가 안전하게 다른 스레드로 이동
    });

    handle.join().unwrap();  // 스레드가 끝날 때까지 대기
}

String 타입은 Send 트레잇을 구현하고 있어, move 키워드를 사용해 새 스레드로 안전하게 전달할 수 있다.

일부 타입은 Send 트레잇을 구현하지 않는다. Rc<T>는 참조 카운팅을 제공하는 타입으로, 여러 스레드에서 공유하는 경우 안전하지 않기 때문에 Send를 구현하지 않는다.

[EX30-6-2]

use std::rc::Rc;
use std::thread;

fn main() {
    let data = Rc::new(5);

    // 컴파일 에러: `Rc`는 `Send`를 구현하지 않음
    let handle = thread::spawn(move || {
        println!("{}", data);
    });

    handle.join().unwrap();
}

Rc<T> 대신 스레드 안전한 참조 카운팅을 제공하는 Arc<T>를 사용하면 해결할 수 있다.

[EX30-6-3]

use std::sync::Arc;
use std::thread;

fn main() {
    let data = Arc::new(5);
    let handle = thread::spawn({
        let data = Arc::clone(&data);  // Arc는 안전하게 복사 가능
        move || {
            println!("{}", data);
        }
    });

    handle.join().unwrap();
}

Sync 트레잇은 여러 스레드에서 참조를 통해 동시 접근이 가능하지만, 안전하게 동작할 수 있는 타입을 나타낸다. 어떤 타입이 Sync를 구현했다면, 그 타입의 참조(&T)는 여러 스레드에서 동시에 접근해도 문제가 없다.

  • 모든 불변 데이터(&T)는 스레드 간에 안전하게 공유될 수 있으므로 Sync를 구현한다.

  • 뮤텍스나 읽기-쓰기 락(Lock) 타입도 Sync를 구현하여 여러 스레드에서 동시 접근이 가능하다.

[EX30-6-4]

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let data = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let data = Arc::clone(&data);
        let handle = thread::spawn(move || {
            let mut num = data.lock().unwrap();
            *num += 1;
        });

        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("Result: {}", *data.lock().unwrap());
}

Mutex<i32>는 Sync 트레잇을 구현하고 있어, 여러 스레드에서 안전하게 공유될 수 있다. Arc는 스레드 안전한 참조 카운팅을 제공하고, Mutex는 뮤텍스 락을 통해 공유 자원의 동시 접근을 가능하게 한다.

Send와 Sync 트레잇의 규칙

Send는 소유권 이동에 중점을 두고, Sync는 동시 참조에 중점을 둔다.

Send: 타입 T가 Send 트레잇을 구현하면, T는 다른 스레드로 이동 가능하다.

Sync: 타입 T가 Sync 트레잇을 구현하면, &T는 여러 스레드에서 공유 가능하다.

대부분의 기본 타입은 자동으로 Send와 Sync를 구현한다. 하지만, 스레드 안전하지 않은 타입(Rc<T>, RefCell<T>, UnsafeCell<T> 등)은 이러한 트레잇을 구현하지 않는다.

스레드 간에 데이터를 이동하려면 Send가 필요하고, 스레드 간에 불변 데이터를 공유하려면 Sync가 필요하다. 안전성을 위해 기본적으로 데이터의 스레드 사용을 제한하며, 스레드 안전한 타입만 Send와 Sync를 구현할 수 있다.

[EX30-6-5]

use std::rc::Rc;
use std::sync::Arc;
use std::thread;

fn main() {
    let data_rc = Rc::new(5);

    // Rc는 Send가 아니므로 스레드 간에 사용할 수 없음
    // thread::spawn(move || println!("{}", data_rc)); // 컴파일 에러
    let data_arc = Arc::new(5);

    thread::spawn({
        let data = Arc::clone(&data_arc);  // Arc는 Send를 구현하므로 스레드 간 이동 가능
        move || println!("{}", data)
    }).join().unwrap();

    // Arc는 Sync를 구현하므로 여러 스레드에서 공유 가능
    let data_sync = Arc::new(5);

    thread::spawn({
        let data = Arc::clone(&data_sync);
        move || println!("Shared data: {}", data)
    }).join().unwrap();
}

Rc<T>는 스레드 간 공유가 안전하지 않기 때문에 Send와 Sync를 구현하지 않는다.

Arc<T>는 스레드 안전한 참조 카운팅을 제공하므로 Send와 Sync를 모두 구현하여 스레드 간에 안전하게 이동 및 공유할 수 있다.

30.7 Channel - Message Passing

채널(Channel)은 스레드 간에 데이터를 전달하기 위한 주요 방법 중 하나이다. 채널은 메시지 전달(message passing) 기반의 동시성(concurrency) 패턴을 구현하며, 공유 메모리 대신 메시지를 통해 데이터를 전달하도록 권장한다. 이는 데이터 소유권과 스레드 안전성을 자연스럽게 보장한다.

표준 라이브러리의 std::sync::mpsc 모듈에서 채널 기능을 제공한다.

  • mpsc: "multiple producer, single consumer"의 약자로, 여러 생산자(Producer)와 하나의 소비자(Consumer)가 있는 채널을 의미한다. Sender와 Receiver로 구성된다.

채널에는 송신자(Sender)와 수신자 (Receiver)가 있다. 송신자에서 데이터와 함께 호출하는 메서드들은 수신자에서 데이터를 받으며 완료된다. 송신자나 수신자둘 중에 하나가 떨어져 버리면 (drop) 채널이 닫혔다고 한다.

[EX30-7-1]

use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    // 채널 생성
    let (tx, rx) = mpsc::channel();

    // 새 스레드에서 메시지 전송
    thread::spawn(move || {
        let messages = vec!["Hello", "from", "the", "thread!"];

        for msg in messages {
            tx.send(msg).unwrap(); // 메시지 전송
            thread::sleep(Duration::from_millis(500)); // 일부러 지연
        }
    });

    // 메시지 수신
    for received in rx {
        println!("Received: {}", received);
    }
}

  • 채널을 통해 데이터를 보낼 때 데이터의 소유권이 이동한다. 이로 인해, 데이터 경합(race condition)이 방지된다.

  • 채널은 기본적으로 비동기적이며, send 호출은 수신자(Receiver)가 준비되지 않아도 진행된다. 버퍼가 초과하지 않는 한 데이터는 안전하게 전송된다.

다중 생산자(Multiple Producers)
Sender를 복제(clone())하여 여러 생산자가 하나의 소비자에게 데이터를 보낼 수 있다.

[EX30-7-2]

use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();
    let tx1 = tx.clone();

    thread::spawn(move || {
        tx1.send("Message from tx1").unwrap();
    });

    thread::spawn(move || {
        tx.send("Message from tx").unwrap();
    });

    for received in rx {
        println!("Received: {}", received);
    }
}

단방향 채널과 동기화

기본 채널은 단방향이지만, 양방향 통신을 구현하려면 두 개의 채널을 사용할 수 있다. 하나는 요청(Request)을 보내고, 다른 하나는 응답(Response)을 받도록 설정한다.

[EX30-7-3]

use std::sync::mpsc;
use std::thread;

fn main() {
    let (req_tx, req_rx) = mpsc::channel();
    let (res_tx, res_rx) = mpsc::channel();

    thread::spawn(move || {
        let request = req_rx.recv().unwrap();

        println!("Worker received: {}", request);
        res_tx.send("Response from worker").unwrap();
    });

    req_tx.send("Request from main").unwrap();

    let response = res_rx.recv().unwrap();
    println!("Main received: {}", response);
}

채널은 간단하고 직관적이지만, 데이터 전송 비용(복사나 이동)이 있을 수 있다. 대안으로 뮤텍스(Mutex)와 아토믹 변수(Atomic)를 활용하거나, crossbeam-channel 크레이트을 사용하는 방법이 있다.

broadcast 채널
이 채널은 mpsc와는 다르게, 데이터를 발행한 후 여러 소비자에게 동일한 데이터를 전달할 수 있다. broadcast 채널은 tokio와 같은 비동기 라이브러리에서 제공되며, tokio::sync::broadcast 모듈에 정의되어 있다.

[dependencies]
tokio = { version = "1", features = ["full"] }

[EX30-7-4]

use tokio::sync::broadcast;
use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    // 채널 생성 (버퍼 크기 16)
    let (tx, _) = broadcast::channel(16);

    // 소비자 1
    tokio::spawn(create_consumer(tx.subscribe(), "Consumer 1"));

    // 소비자 2
    tokio::spawn(create_consumer(tx.subscribe(), "Consumer 2"));

    // 메시지를 계속 보내기 위한 루프
    loop {
        // 수신자가 메시지를 받을 준비가 되었는지 확인한 후 메시지 보내기
        match tx.send("Hello to all consumers") {
            Ok(_) => println!("Message sent to consumers."),
            Err(e) => {
                eprintln!("Error sending message: {:?}", e);
                // SendError가 발생하면 해당 메시지를 수신할 수 있는 소비자가 없다는 의미
            }
        }

        // 2초마다 메시지를 보내도록 설정
        sleep(Duration::from_secs(2)).await;
    }
}

async fn create_consumer(mut rx: broadcast::Receiver<&str>, name: &str) {
    loop {
        match rx.recv().await {
            Ok(msg) => println!("{} received: {}", name, msg),
            Err(e) => {
                eprintln!("{} error: {:?}", name, e);
                // 수신자에게 더 이상 메시지를 보낼 수 없을 때 계속 대기
                // 잠시 대기 후 재시도
                sleep(Duration::from_secs(1)).await;
            }
        }
    }
}

경계(Bounded)와 무경계(Unbounded) 채널

Rust 표준 라이브러리에서 "경계 채널"과 "무경계 채널"이라는 용어는 공식적으로 정의된 개념은 아니다.

(1) 경계 채널 (Bounded Channel)

경계 채널은 버퍼 크기가 정해져 있는 채널이다. 즉, 채널에 일정한 수의 메시지만 저장할 수 있다. 버퍼가 가득 차면, 생산자가 메시지를 보내기 전에 소비자가 메시지를 소비할 때까지 대기하게 된다.

  • 채널이 가득 차면 송신자는 대기해야 한다.

  • 생산자와 소비자가 동시에 작업을 할 수 있지만, 버퍼가 차면 생산자는 차례를 기다려야 한다.

  • tokio::sync::mpsc::channel(n)을 사용할 때, n은 버퍼 크기를 나타내며, 이 값이 경계 채널의 크기를 결정한다.

  • 생산자가 너무 많은 메시지를 보낼 때, 소비자가 메시지를 소비할 수 있는 속도에 맞춰 메시지의 흐름을 제어할 수 있다.

[EX30-7-5]

use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::sync_channel(2); // 버퍼 크기 2

    thread::spawn(move || {
        tx.send(1).unwrap();
        tx.send(2).unwrap();
        tx.send(3).unwrap(); // 이 시점에서 채널이 가득 차면 대기
    });

    for _ in 0..3 {
        println!("{}", rx.recv().unwrap()); // 메시지를 소비
    }
}

(2) 무경계 채널 (Unbounded Channel)

무경계 채널은 버퍼 크기가 제한이 없는 채널이다. 이 채널은 메시지의 수에 제한이 없으며, 생산자는 메시지를 보내는 대로 소비자가 소비할 수 있다. 무경계 채널에서는 버퍼가 가득 차서 대기할 필요가 없지만, 시스템 자원이나 성능 문제로 메시지가 너무 많이 쌓일 수 있다.

  • 생산자가 메시지를 보내면 소비자는 메시지를 소비할 수 있을 때까지 계속해서 수신한다.

  • 생산자가 빠르게 메시지를 보내고 소비자가 이를 처리하는 속도에 관계없이 메시지를 계속 전송할 수 있다.

  • tokio::sync::mpsc::unbounded_channel()을 사용하여 무경계 채널을 생성할 수 있다.

  • 무경계 채널은 시스템의 성능에 영향을 미칠 수 있기 때문에 너무 많은 메시지가 쌓이지 않도록 관리가 필요하다.

[EX30-7-6]

use tokio::sync::mpsc;

#[tokio::main]
async fn main() {
    let (tx, mut rx) = mpsc::unbounded_channel();

    // 메시지 보내기
    tx.send("Hello").unwrap();
    tx.send("World").unwrap();

    // 메시지 받기
    while let Some(message) = rx.recv().await {
        println!("Received: {}", message);
    }
}

30.8 crossbeam-channel

crossbeam-channel은 고성능 채널 기능을 제공하는 크레이트이다. 표준 라이브러리의 std::sync::mpsc 채널보다 더 유연하고 효율적인 메시지 전달 메커니즘을 제공한다. 고성능을 요구하는 다중 소비자(multiple consumers) 및 멀티프로듀서-멀티컨슈머(MPMC) 패턴에 적합하다.

  • 멀티프로듀서-멀티컨슈머(MPMC) 지원: 여러 Sender와 여러 Receiver가 동시에 작동.

  • 메시지 전달 속도가 더 빠르며, 대기 중인 스레드 관리가 더 효율적이다.

  • 무제한(unbounded), 고정 크기(fixed size), 제로 크기(zero-sized) 버퍼를 지원.

  • 표준 라이브러리와 비슷한 API 제공

[dependencies]
crossbeam-channel = "0.5"

단일 생산자-소비자 (Single Producer, Single Consumer)

[EX30-8-1]

use crossbeam_channel::unbounded;

fn main() {

작업 풀(Thread Pool)을 구현하면, 작업을 여러 스레드로 효율적으로 분산할 수 있다. 작업 풀은 생산자(Producer)가 작업을 큐에 추가하고, 여러 워커(Worker) 스레드가 이를 병렬로 처리하는 구조로 동작한다.

  • Sender와 Receiver: crossbeam-channel로 작업을 전달.

  • 워커 스레드: 각 스레드가 Receiver를 통해 작업을 받는다.

  • Graceful Shutdown: 작업이 모두 끝났을 때 워커를 정리한다.

[EX30-8-6]

4개의 스레드가 8개의 작업을 병렬 처리한다.

use crossbeam_channel::{bounded, Sender};
use std::thread;

struct ThreadPool {
    workers: Vec<thread::JoinHandle<()>>,
    sender: Sender<Box<dyn FnOnce() + Send + 'static>>,
}

impl ThreadPool {
    /// 새로운 스레드 풀 생성
    pub fn new(size: usize) -> Self {
        assert!(size > 0);

        // 작업을 주고받는 채널 생성
        let (sender, receiver) = bounded::<Box<dyn FnOnce() + Send + 'static>>(size);

        // 채널의 수신자를 여러 스레드가 공유
        let receiver = std::sync::Arc::new(receiver);
        let mut workers = Vec::with_capacity(size);

        // 워커 스레드 생성
        for _ in 0..size {
            let receiver_clone = std::sync::Arc::clone(&receiver);
            let handle = thread::spawn(move || {
                while let Ok(task) = receiver_clone.recv() {
                    // 수신된 작업 실행
                    task();
                }
            });

            workers.push(handle);
        }

        ThreadPool { workers, sender }
    }

    /// 작업을 스레드 풀에 추가
    pub fn execute<F>(&self, task: F)
    where
        F: FnOnce() + Send + 'static,
    {
        self.sender.send(Box::new(task)).unwrap();
    }
}

impl Drop for ThreadPool {
    /// 스레드 풀 종료 시 모든 스레드 정리
    fn drop(&mut self) {
        drop(&self.sender); // 채널 닫기

        // 모든 워커가 종료될 때까지 대기
        for worker in self.workers.drain(..) {
            worker.join().unwrap();
        }
    }
}

fn main() {
    // 스레드 풀 생성 (스레드 4개)
    let pool = ThreadPool::new(4);

    // 작업 추가
    for i in 0..8 {
        pool.execute(move || {
            println!("Task {} is running on thread {:?}", i, thread::current().id());

            std::thread::sleep(std::time::Duration::from_millis(500));
        });
    }

    println!("All tasks have been assigned.");
} // drop 호출로 모든 스레드 종료

30.9 상태 공유(Shared-State)

여러 스레드나 코드 블록에서 동일한 데이터를 안전하게 접근하고 수정할 수 있도록 하는 방법을 의미한다. 채널을 이용한 방법이 값에 대해 단일 소유권을 가지는 것이라면 메모리를 공유하는 것은 동시에 하나의 메모리 주소에 대해 여러 소유권을 갖게 되는 것을 의미한다. 스레드 사이의 안전한 데이터 공유를 위해 Arc와 Mutex를 사용한다.

Mutex(mutual exclusion)는 특정 시점에 오직 하나의 스레드만 데이터에 접근하는 것을 허용한다. 뮤텍스 안에서 데이터를 접근하면 접근 요청을 먼저 요구한 시그널이 뮤텍스 락을 요청한다. 이 락은 뮤텍스가 데이터에 대한 상호 배타적 접근을 가능하게 하는 자료구조이다. 따라서 뮤텍스는 이러한 잠금 시스템을 통해 데이터를 보호한다.

이 코드는 여러 스레드를 생성하여 counter 값을 증가시키는 예제이다. 루프를 돌면서 10개의 스레드를 생성하고 모든 스레드에 동일한 클로저를 제공한다.

[EX30-9-1]

use std::thread;
use std::sync::Mutex;

fn main() {
    let counter = Mutex::new(0);
    let mut handles = vec\![\];

    for _ in 0..10 {
        let handle = thread::spawn(move|| {
            let mut num = counter.lock().unwrap();
            *num += 1;
        });

        handles.push(handle);
    }

    for handle in handles{
        handle.join().unwrap();
    }

    println!("Result: {}", *counter.lock().unwrap());
}

Arc<T>는 Rc<T>타입과 유사한 기능을 하면서 동시성 요소를 구현하고 있어 스레드 간 사용해도 안전한 타입이다. 다른 타입이 원자적(Atomic)으로 구현되지 않은 이유는 해당 타입 사용이 성능 이슈를 가져올 수 있기 때문이다. 단일 스레드에서는 원자성을 강제하지 않음으로 더 빠르게 코드를 실행할 수 있다.

[EX30-9-2]

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for i in 0..10 {
        let counter = Arc::clone(&counter);
        let handle = thread::spawn(move|| {
            let mut num = counter.lock().unwrap();
            println!("{} th thread",i);
            *num+=1;
        });

        println!("i traverse {} ",i);

        handles.push(handle);
    }

    for handle in handles {
        println!("in handle");

        handle.join().unwrap();
        println!("Result: {}", *counter.lock().unwrap());
    }
}

비동기 런타임으로 많이 사용되는 Tokio와 같은 비동기 실행 환경에서는 동기화 메커니즘도 비동기적으로 작동해야 한다. Tokio의 Mutex를 사용하면, 다중 스레드 환경의 비동기 문맥에서도 안전한 상태 공유가 가능하다.

일반적인 동기화 도구(std::sync::Mutex)는 태스크가 락을 기다리는 동안 스레드를 차단하지만, tokio::sync::Mutex는 비동기적으로 락을 대기하여 런타임이 다른 태스크를 실행할 수 있도록 한다.

[EX30-9-3]

use std::sync::Arc;
use tokio::sync::Mutex;

#[tokio::main]
async fn main() {
    let data = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let data_clone = Arc::clone(&data);
        let handle = tokio::spawn(async move {
            let mut num = data_clone.lock().await;
            *num += 1;
        });

        handles.push(handle);
    }

    for handle in handles {
        handle.await.unwrap();
    }

    println!("Final value: {}", *data.lock().await);
}

상태 공유 시 주의할 점:

  • Mutex를 사용할 때, 락을 해제하지 않으면 데드락 발생할 수 있다.

  • Rust는 범위 기반으로 락을 자동 해제하여 이를 방지한다.

  • Mutex는 락-언락 과정에서 오버헤드가 발생한다.

  • 읽기가 많은 경우 RwLock이 더 적합하다.

  • 비동기 코드에서 동기 Mutex를 사용할 경우 실행이 차단(Block)될 수 있으므로 tokio::sync::Mutex를 사용해야 한다.

공유 상태 관리는 안전성과 성능을 균형 있게 제공하며, 필요에 따라 적합한 도구(Arc, Mutex, RwLock)를 선택할 수 있다.

30.10 AtomicUsize

std::sync::atomic 모듈에서 제공하는 원자적(atomic) 타입 중 하나로, usize 타입의 값을 여러 스레드에서 안전하게 다룰 수 있도록 한다. 원자적 연산을 통해 멀티스레드 환경에서 데이터 경합(race condition)을 방지하고, 일관성 있는 값을 유지할 수 있게 해준다.

  • 여러 스레드가 동시에 값을 읽거나 쓸 때, 경합을 방지하고, 스레드 안전성을 보장한다.

  • 제공하는 메서드를 통해, 값의 증가, 감소 등의 연산을 원자적으로 처리한다.

  • 연산의 순서와 동작을 제어할 수 있는 Ordering 매개변수를 제공한다.

AtomicUsize는 보통 멀티스레딩 환경에서 동기화를 위해 사용된다. 예를 들어, 여러 스레드가 같은 카운터를 동시에 증가시키는 경우에 유용한다. AtomicUsize는 fetch_add, fetch_sub, load, store 등의 메서드를 통해 원자적인 작업을 제공한다.

[EX30-10-1]

use std::sync::atomic::{AtomicUsize, Ordering};
use std::sync::Arc;
use std::thread;

fn main() {
    let counter = Arc::new(AtomicUsize::new(0));  // AtomicUsize로 카운터 생성
    let mut handles = vec![];

    // 10개의 스레드에서 원자적으로 값을 증가시킴
    for _ in 0..10 {
        let counter_clone = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            for _ in 0..100 {
                counter_clone.fetch_add(1, Ordering::SeqCst);  // 원자적 증가
            }
        });

        handles.push(handle);
    }

    // 모든 스레드가 종료될 때까지 대기
    for handle in handles {
        handle.join().unwrap();
    }

    // 최종 카운트를 출력
    println!("Final count: {}", counter.load(Ordering::SeqCst));
}

메모리 순서 (Ordering): AtomicUsize의 원자적 연산에서 중요한 부분은 메모리 순서이다. Ordering은 각 원자적 연산이 다른 연산과 어떻게 상호작용할지를 결정한다. 메모리 순서를 설정함으로써 스레드 간의 연산 순서를 제어할 수 있다.

  • Ordering::Relaxed: 메모리 순서를 보장하지 않는다. 가장 빠른 방식으로 동작하지만, 다른 스레드에서의 읽기/쓰기 순서에 대한 보장이 없다.

  • Ordering::Acquire: 현재 연산이 완료된 후, 그 이전 연산이 완료될 때까지 기다린다. 다른 스레드에서의 읽기/쓰기 순서를 보장한다.

  • Ordering::Release: 현재 연산이 완료된 후, 그 이후 연산이 진행된다. 다른 스레드에서의 읽기/쓰기 순서를 보장한다.

  • Ordering::AcqRel: Acquire와 Release를 모두 적용한 방식이다.

  • Ordering::SeqCst: 가장 강력한 보장으로, 모든 연산이 순차적 일관성을 유지하도록 보장한다.

[EX30-10-2]

//AtomicUsize와 Ordering의 예시
use std::sync::atomic::{AtomicUsize, Ordering};
use std::sync::Arc;
use std::thread;

fn main() {
    let atomic_val = Arc::new(AtomicUsize::new(0));
    let handles: Vec<_> = (0..10).map(|_| {
        let atomic_val = Arc::clone(&atomic_val);

        thread::spawn(move || {
            atomic_val.fetch_add(1, Ordering::SeqCst);
        })
    }).collect();

    for handle in handles {
        handle.join().unwrap();
    }

    println!("Final value: {}", atomic_val.load(Ordering::SeqCst));
}

위 코드에서는 여러 스레드가 fetch_add를 사용하여 AtomicUsize 값을 안전하게 증가시킨다. 각 스레드는 원자적으로 값을 증가시키며, 메모리 순서가 SeqCst로 설정되어 있기 때문에 순차적 일관성을 보장한다.

아래는 AtomicUsize의 주요 메서드 기능을 설명한 내용이다.

메서드 설명
new(value: usize) -> AtomicUsize 새로운 AtomicUsize 인스턴스를 생성. 초기값을 설정해야 한다.
load(order: Ordering) -> usize 원자적으로 현재 값을 읽어온다. 메모리 순서를 지정해야 하며, 이는 읽기 작업의 동기화를 제어한다.
store(value: usize, order: Ordering) 원자적으로 값을 설정. 메모리 순서를 지정해야 하며, 이는 쓰기 작업의 동기화를 제어한다.
swap(value: usize, order: Ordering) -> usize 현재 값을 새 값으로 원자적으로 교체하고, 이전 값을 반환한다.
compare_exchange(current: usize, new: usize, success: Ordering, failure: Ordering) -> Result<usize, usize> 현재 값이 current와 같다면 새 값으로 교체하고 Ok(이전 값)을 반환한다. 실패하면 Err(현재 값)을 반환한다.
compare_exchange_weak(current: usize, new: usize, success: Ordering, failure: Ordering) -> Result<usize, usize> compare_exchange와 유사하지만, 실패 시 더 가벼운 검사를 수행하며 루프에서 사용하기 적합하다.
fetch_add(value: usize, order: Ordering) -> usize 현재 값에 value를 더하고, 이전 값을 반환한다.
fetch_sub(value: usize, order: Ordering) -> usize 현재 값에서 value를 빼고, 이전 값을 반환한다.
fetch_and(value: usize, order: Ordering) -> usize 현재 값과 value의 비트 AND 연산을 수행하고, 이전 값을 반환한다.
fetch_or(value: usize, order: Ordering) -> usize 현재 값과 value의 비트 OR 연산을 수행하고, 이전 값을 반환한다.
fetch_xor(value: usize, order: Ordering) -> usize 현재 값과 value의 비트 XOR 연산을 수행하고, 이전 값을 반환한다.
fetch_update<F>(&self, set_order: Ordering, fetch_order: Ordering, f: F) -> Result<usize, usize> 클로저 f를 사용하여 새 값을 계산한 뒤, 조건이 충족되면 원자적으로 값을 업데이트한다. 실패 시 Err(현재 값)을 반환한다.
into_inner(self) -> usize AtomicUsize를 해제하여 내부 값을 반환한다.
get_mut(&mut self) -> &mut usize 내부 값을 직접 변경할 수 있는 가변 참조를 반환한다. (스레드 안전이 필요하지 않을 때 사용)

값 증가:

use std::sync::atomic::{AtomicUsize, Ordering};

let counter = AtomicUsize::new(0);

counter.fetch_add(1, Ordering::SeqCst);

println!("Counter: {}", counter.load(Ordering::SeqCst));

조건부 업데이트:

use std::sync::atomic::{AtomicUsize, Ordering};

let value = AtomicUsize::new(42);
let result = value.compare_exchange(42, 100, Ordering::SeqCst, Ordering::SeqCst);

match result {
    Ok(prev) => println!("Updated from {} to 100", prev),
    Err(current) => println!("Failed to update, current value is {}", current),
}

클로저 기반 업데이트:

use std::sync::atomic::{AtomicUsize, Ordering};

let value = AtomicUsize::new(5);
let result = value.fetch_update(Ordering::SeqCst, Ordering::SeqCst, |x| {
    if x < 10 {
        Some(x * 2)
    } else {
        None
    }
});

match result {
    Ok(prev) => println!("Updated from {} to {}", prev, value.load(Ordering::SeqCst)),
    Err(current) => println!("Update failed, current value is {}", current),
}

30.11 Condvar

Condvar는 조건 변수를 이용하여 스레드 간 통신을 처리할 때 사용된다. Condvar는 주로 하나의 스레드가 특정 조건을 기다리는 동안 대기하고, 다른 스레드가 그 조건을 만족시킬 때 신호를 보내어 깨어나게 하는 데 사용된다.

Condvar는 뮤텍스(Mutex)와 함께 사용된다. 이를 통해 여러 스레드가 안전하게 공유 데이터를 기다리거나 조건을 충족시킬 수 있다.

  • wait: 스레드가 조건 변수를 기다리게 한다. 이 메서드는 뮤텍스 락을 해제하고 스레드를 대기 상태로 만든다.

  • notify_one: 대기 중인 스레드 하나를 깨운다.

  • notify_all: 대기 중인 모든 스레드를 깨운다.

Condvar와 Mutex를 사용하여 생산자-소비자 패턴을 구현한다. 한 스레드가 작업을 준비하면 다른 스레드가 작업을 시작하도록 신호를 보내는 방식이다. 조건이 만족될 때까지 무한정 대기하는 것이 아니라, 일정 시간 동안만 대기한 후 타임아웃이 발생하면 대기에서 빠져나오도록 할 수 있다. 이를 위해 wait_timeout 메서드를 사용한다.

[EX30-11-1]

use std::sync::{Arc, Mutex, Condvar};
use std::thread;
use std::time::Duration;

fn main() {
    let pair = Arc::new((Mutex::new(false), Condvar::new()));
    let pair2 = Arc::clone(&pair);
    let consumer = thread::spawn(move || {
        let (lock, cvar) = &*pair2;
        let mut started = lock.lock().unwrap();
        let timeout_result = cvar
            .wait_timeout(started, Duration::from_secs(3))
            .unwrap();

        started = timeout_result.0;

        if *started {
            println!("작업 시작!");
        } else {
            println!("타임아웃 발생, 조건이 만족되지 않음");
        }
    });

    let producer = thread::spawn(move || {
        let (lock, cvar) = &*pair;
        thread::sleep(Duration::from_secs(1)); // 생산 지연
        
        let mut started = lock.lock().unwrap();
        *started = true;
        cvar.notify_one(); // 조건 만족 신호
    });

    consumer.join().unwrap();
    producer.join().unwrap();
}

  1. pair는 Mutex와 Condvar를 포함하는 튜플이다.

  2. Mutex는 공유 상태(여기서는 false로 초기화)를 보호하고, Condvar는 상태 변화에 대한 신호를 기다리기 위해 사용된다.

  3. Arc를 사용하여 여러 스레드에서 안전하게 접근할 수 있도록 설정한다.

  4. 소비자는 Condvar를 통해 공유 상태가 변경되기를 기다린다.

  5. wait_timeout 메서드를 사용하여 최대 3초 동안 조건을 기다린다.

  6. 조건이 충족되었는지 확인한 후, 상태를 출력한다.

  7. 생산자가 조건을 만족시키면 소비자는 "작업 시작!"을 출력한다. 타임아웃이 발생하지 않는다.

notify_all을 사용하면 대기 중인 모든 스레드를 깨울 수 있다. 여러 스레드가 동일한 조건을 기다리고 있을 때 유용하다.

[EX30-11-2]

use std::sync::{Arc, Mutex, Condvar};
use std::thread;
use std::time::Duration;

fn main() {
    let pair = Arc::new((Mutex::new(false), Condvar::new()));

    // 여러 소비자 스레드
    let consumer1 = {
        let pair2 = Arc::clone(&pair);

        thread::spawn(move || {
            let (lock, cvar) = &*pair2;
            let mut started = lock.lock().unwrap();

            while !*started {
                started = cvar.wait(started).unwrap();
            }
            println!("작업 시작1!");
        })
    };

    let consumer2 = {
        let pair3 = Arc::clone(&pair);

        thread::spawn(move || {
            let (lock, cvar) = &*pair3;
            let mut started = lock.lock().unwrap();

            while !*started {
                started = cvar.wait(started).unwrap();
            }

            println!("작업 시작2!");
        })
    };

    // 생산자 스레드 (조건을 만족시킴)
    let producer = thread::spawn({
        let pair2 = Arc::clone(&pair);

        move || {
            thread::sleep(Duration::from_secs(1)); // 생산 지연
            let (lock, cvar) = &*pair2;
            let mut started = lock.lock().unwrap();
            *started = true; // 조건을 만족시키기 위해 true로 설정
            cvar.notify_all(); // 모든 소비자에게 신호 보내기
        }
    });

    consumer1.join().unwrap();
    consumer2.join().unwrap();
    producer.join().unwrap();
}

30.12 데이터 병렬 처리(Data Parallelism)

Rayon은 데이터 병렬 처리를 위한 라이브러리로, 간편하게 멀티스레드 병렬 처리를 구현할 수 있게 도와준다. Rayon 크레이트는 반복자 기반으로 병렬 처리를 지원하여, 일반적인 순차적 반복자 연산을 병렬 연산으로 변환할 수 있다.

  • .iter() 대신 .par_iter()를 사용하여 반복자의 작업을 병렬로 실행할 수 있다.

  • 내부적으로 작업을 자동으로 분할하고 스레드 풀을 관리하여, 안전하게 병렬 처리를 수행한다.

  • 기존의 반복자 기반 코드를 수정하지 않고 병렬화 할 수 있다.

[dependencies]
rayon = "1.7"

par_iter를 통한 병렬 처리

벡터의 모든 요소를 제곱하는 작업을 병렬로 처리한다.

[EX30-12-1]

use rayon::prelude::*;

fn main() {
    let numbers: Vec<i32> = (1..100).collect();

    // 순차적 처리
    let squares_sequential: Vec<i32> = numbers.iter().map(|&x| x * x).collect();
   
    // 병렬 처리
    let squares_parallel: Vec<i32> = numbers.par_iter().map(|&x| x * x).collect();

    println!("Sequential: {:?}", squares_sequential);
    println!("Parallel: {:?}", squares_parallel);
}

  1. par_iter()를 사용하면 iter()와 동일한 방식으로 벡터를 순회하면서 병렬로 처리한다.

  2. map() 연산이 여러 스레드에서 동시에 실행된다.

  3. collect()는 병렬 처리된 결과를 다시 벡터로 모아준다.

par_iter_mut로 병렬 가변 처리

벡터의 값을 직접 변경해야 하는 경우, par_iter_mut()를 사용해 각 요소를 병렬로 수정할 수 있다.

[EX30-12-2]

use rayon```::prelude::*;

fn main() {
    let mut numbers: Vec<i32> = (1..100).collect();

    // 벡터의 값을 병렬로 수정
    numbers.par_iter_mut().for_each(|x| {
        *x *= 2;
    });

    println!("Modified: {:?}", numbers);
}

  1. par_iter_mut()를 통해 벡터의 가변 참조를 얻어, 요소를 병렬로 수정한다.

  2. for_each()를 사용하여 각 요소를 병렬로 순회하면서 값을 변경한다.

par_sort를 사용한 병렬 정렬

벡터의 정렬 작업도 병렬로 수행할 수 있다. 대용량 데이터를 정렬할 때 성능 향상이 매우 크다.

[EX30-12-3]

use rayon::prelude::*;

fn main() {
    let mut numbers = vec![5, 3, 8, 6, 1, 9, 2, 7, 4];

    // 병렬 정렬
    numbers.par_sort();
    println!("Sorted: {:?}", numbers);
}

  1. par_sort()를 사용하여 벡터를 병렬로 정렬한다.

  2. 내부적으로 빠른 정렬 알고리즘을 여러 스레드를 활용해 병렬로 처리한다.

par_chunks를 사용한 병렬 청크 처리

par_chunks 메서드를 사용하면 데이터를 여러 청크로 나누어 병렬로 처리할 수 있다.

[EX30-12-4]

use rayon::prelude::*;

fn main() {
    let numbers: Vec<i32> = (1..100).collect();

    // 벡터를 청크 단위로 병렬 처리
    numbers.par_chunks(10).for_each(|chunk| {
        println!("Chunk: {:?}", chunk);
    });
}

  1. par_chunks()는 데이터를 일정 크기의 청크로 나누어 병렬로 처리한다.

  2. 각 청크를 병렬로 처리하여 작업을 나눌 수 있다.

병렬 반복 작업: reduce와 fold

병렬 축소 작업도 지원한다. 벡터의 모든 요소를 합산하는 작업을 병렬로 수행할 수 있다.

[EX30-12-5]

use rayon::prelude::*;

fn main() {
    let numbers: Vec<i32> = (1..100).collect();

    // 병렬로 요소들의 합을 계산
    let sum: i32 = numbers.par_iter().cloned().reduce(|| 0, |a, b| a + b);

    println!("Sum: {}", sum);
}

  • reduce는 병렬로 데이터를 축소(예: 합산)하는 작업을 수행한다.

  • || 0은 초기값이며, |a, b| a + b는 합산할 때 사용하는 클로저이다.

병렬로 폴드(fold)

fold는 상태를 유지하면서 반복 작업을 처리할 때 유용하다. 병렬화 된 작업에서도 이를 사용하여 보다 복잡한 상태 기반 처리를 수행할 수 있다.

[EX30-12-6]

use rayon::prelude::*;

fn main() {
    let numbers: Vec<u64> = (1..=20).collect();

    // 병렬 폴드를 사용하여 모든 요소를 곱함
    let product: u64 = numbers.par_iter()
        .cloned()
        .fold(|| 1, |a, b| a * b)
        .reduce(|| 1, |a, b| a * b);

    println!("Product: {}", product);
}

출력 값은 20!로, 모든 숫자를 곱한 결과값이다.

  1. numbers.par_iter():
    벡터 numbers에 대해 병렬 반복을 시작한다. 이 메서드는 벡터를 여러 스레드에서 병렬로 처리할 수 있게 한다.

  2. cloned():
    par_iter()는 참조를 반환하므로, cloned()를 사용하여 소유권을 이동시키고 각 요소를 i64 타입으로 복사한다.

  3. fold(|| 1, |a, b| a * b):
    fold()는 병렬 작업에서 각 스레드가 데이터를 처리하도록 하고, 각 스레드 내에서 곱셈을 수행한다. 첫 번째 인자 || 1은 각 스레드에서의 초기값을 설정한다.

  4. 두 번째 인자 |a, b| a * b는 각 스레드가 계산하는 방식이다. 각 요소를 곱하는 방식으로 연산이 수행된다.

  5. reduce(|| 1, |a, b| a * b):
    reduce()는 병렬로 처리된 결과들을 결합하는 역할을 한다. fold()로 얻은 부분 결과들을 병합한다. 병렬 작업이 끝나면, 최종 결과를 하나로 모은다.

scope로 병렬 작업 그룹 관리

scope 기능을 제공하여 서로 다른 병렬 작업을 쉽게 관리하고, 병렬 작업 간의 상호 참조도 안전하게 처리할 수 있다.

[EX30-12-7]

use rayon::scope;

fn main() {
    scope(|s| {
        s.spawn(|_| {
            println!("Task 1");
        });

        s.spawn(|_| {
            println!("Task 2");
        });
    });

    println!("Main thread done");
}

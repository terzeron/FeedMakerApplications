[Kubernetes 스터디] service 란?

@chubbyarnold
2026년 1월 14일 2:55 오후

Deployment와 Service

쿠버네티스를 공부하면서 가장 기본이면서도 중요하다고 느낀 디플로이먼트(Deployment) 와 외부 통신을 위한 서비스(Service) 3가지 종류에 대해 정리해보려고 합니다.

처음엔 "파드(Pod)만 띄우면 끝 아닌가?"라고 생각했는데, 파드는 언제든 죽었다 살아날 수 있어서 이를 관리해주는 녀석(Deployment)과 고정된 접속 주소를 제공하는 녀석(Service)이 꼭 필요하더라고요.

YAML 파일과 터미널 실습 로그를 보면서 하나씩 뜯어봅시다!

1. 파드 공장장: 디플로이먼트 (Deployment)

가장 먼저 등장하는 건 Deployment입니다. 이건 쉽게 말해 "파드 관리자" 예요.

개념: 우리가 파드를 하나하나 수동으로 관리하면 너무 힘들잖아요? 디플로이먼트는 우리가 "파드 3개 유지해줘!"라고 주문서를 넣으면, 알아서 3개를 딱 맞춰서 실행하고 유지해주는 공장장 역할을 합니다.


apiVersion: apps/v1
kind: Deployment
metadata:
  name: http-go-deployment
spec:
  replicas: 3               # 핵심: "복제본(파드) 3개를 유지해라!"
  selector:
    matchLabels:
      app: http-go          # "app: http-go" 라벨이 붙은 애들을 내 부하로 관리하겠다.
  template:                 # 여기서부터 파드를 찍어낼 틀(Template)
    metadata:
      labels:
        app: http-go
    spec:
      containers:
      - name: http-go
        image: gasbugs/http-go:latest
        ports:
        - containerPort: 8080 # 컨테이너 내부 포트

포인트: replicas: 3 설정 덕분에 잠시 후 터미널에서 파드가 3개 생성된 것을 볼 수 있습니다.

2. 연결의 마법사: 서비스 (Service) 3대장

파드가 잘 떠 있어도, 외부에서 접속하려면 서비스(Service) 가 필요합니다. 파드 IP는 재시작할 때마다 바뀌기 때문이죠. 서비스는 변하지 않는 '고정 접속 경로'를 제공합니다.

① ClusterIP (우리끼리만!)
개념: 쿠버네티스 클러스터 내부에서만 쓸 수 있는 전용 회선입니다. 외부에서는 접근 불가능하고, 내부의 다른 파드들이 이 서비스를 찾을 때 씁니다. (별도 설정 없으면 이게 기본값입니다.)

# ... (생략)
kind: Service
metadata:
  name: http-go-cip
spec:
  selector:
    app: http-go    # 디플로이먼트가 만든 그 파드들로 연결!
  ports:
    - protocol: TCP
      port: 80      # 서비스 대표 포트
      targetPort: 8080 # 실제 파드 포트

② NodePort (구멍 뚫기!)
개념: 클러스터의 모든 노드(서버)에 똑같은 문(Port)을 뚫어버리는 방식입니다. 외부에서 노드IP:포트번호로 접속할 수 있습니다.

# ... (생략)
kind: Service
metadata:
  name: http-go-np 
spec:
  type: NodePort      # 타입 명시 필수!
  selector:
    app: http-go
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30001 # 30000~32767 사이 포트 고정 가능

해석: 외부에서 어떤_노드_IP:30001로 들어오면 파드로 연결해줍니다.

③ LoadBalancer (클라우드의 힘!)
개념: AWS, GCP 같은 클라우드 환경에서 진짜 외부 접속용 공인 IP를 받아오는 방식입니다. 실무에서 외부 서비스를 할 때 가장 많이 사용합니다.

# ... (생략)
kind: Service
metadata:
  name: http-go-lb 
spec:
  type: LoadBalancer # 클라우드 로드밸런서 요청
  selector:
    app: http-go
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30002

3. 실습 로그 분석

(1) 파드 확인 (Running)

kubectl get pod 

NAME                                   READY   STATUS    RESTARTS   AGE
http-go-deployment-xxxxxxxxx-xxxxx     1/1     Running   0          27s

(2) LoadBalancer IP 할당 과정

[첫 번째 조회: 32초 경과]

kubectl get svc

NAME          TYPE           CLUSTER-IP      EXTERNAL-IP   PORT(S)
http-go-lb    LoadBalancer   10.xxx.xxx.xx   <pending>     80:30002/TCP

http-go-lb의 EXTERNAL-IP가 <pending> 상태입니다. 클라우드 제공자(GCP)에게 "IP 하나만 주세요~" 하고 요청 후 기다리는 중인 거죠.

[두 번째 조회: 44초 경과]

kubectl get svc

NAME          TYPE           CLUSTER-IP      EXTERNAL-IP      PORT(S)
http-go-lb    LoadBalancer   10.xxx.xxx.xx   35.xxx.xxx.xxx   80:30002/TCP

드디어! 35.xxx.xxx.xxx 라는 외부 IP가 할당되었습니다. 이제 전 세계 어디서든 저 IP로 접속하면 우리 파드로 연결됩니다!

글로만 보면 헷갈릴 수 있어서 간단한 구조도를 그려봤습니다.

LoadBalancer가 외부 문지기 역할을 하고, 트래픽을 받아서 노드(Node) 로 넘기고, 서비스가 라벨(app: http-go) 을 보고 적절한 Pod로 연결해줍니다.

NodePort vs LoadBalancer

Tomcat(톰캣) 을 쿠버네티스에 배포하고, 외부에서 접속하는 두 가지 방법(NodePort와 LoadBalancer)을 직접 실습해 본 내용을 정리한다.

1. Tomcat 파드 띄우기

방법 1 (NodePort): 내 컴퓨터(Node)의 30005번 문을 열어서 접속하기.
방법 2 (LoadBalancer): 전용 공인 IP를 받아서 80번 포트로 우아하게 접속하기.

2. 배포 파일 분석 (tomcat-deployment.yaml)

1) Tomcat Deployment (파드 생성기)


apiVersion: apps/v1
kind: Deployment
metadata:
  name: tomcat-deployment
spec:
  replicas: 1
  # ... (생략)
    spec:
      containers:
      - name: tomcat
        image: gasbugs21c/tomcat:latest # 톰캣 이미지를 다운로드
        ports:
        - containerPort: 8080            # 톰캣은 원래 8080 포트를 쓴다!

gasbugs21c/tomcat:latest라는 이미지를 가져와서 컨테이너를 1개 실행한다. 톰캣은 태생적으로 8080번 포트에서 일하기 때문에 containerPort를 8080으로 명시해 주었다.

2) Service 1: NodePort (창문 열기)


apiVersion: v1
kind: Service
metadata:
  name: tomcat-nodeport
spec:
  type: NodePort       # 중요: 노드의 포트를 열겠다 선언
  selector:
    app: tomcat
  ports:
    - port: 8080       # 서비스 내부 포트
      targetPort: 8080 # 파드의 실제 포트 (위의 containerPort와 일치해야 함)
      nodePort: 30005  # 외부에서 접속할 문 번호 (고정)

왜 nodePort: 30005를 썼을까?: 보통은 쿠버네티스가 30000~32767 사이에서 아무거나 골라주지만, 우리가 실습을 위해 "30005번으로 들어올게!" 라고 강제로 지정한 것이다.

3) Service 2: LoadBalancer (정문 만들기)


apiVersion: v1
kind: Service
metadata:
  name: tomcat-loadbalancer
spec:
  type: LoadBalancer   # 중요: 클라우드 로드밸런서 요청
  selector:
    app: tomcat
  ports:
    - port: 80         # 외부에는 80번(기본 웹 포트)으로 보이게 함
      targetPort: 8080 # 실제 파드는 8080임
      nodePort: 30006  # 로드밸런서도 내부적으로는 노드포트를 쓴다 (여기선 30006 지정)

특징: 외부 사용자에게는 80번 포트를 노출하고, 내부적으로는 8080번으로 연결해 준다. 사용자가 굳이 :8080을 뒤에 붙이지 않아도 접속되게 하기 위함이다.

  1. 방화벽(Firewall)

쿠버네티스 설정만으로는 외부에서 접속이 안 된다. 왜냐하면 구글 클라우드(GCP) 자체가 기본적으로 외부의 침입을 막고 있기 때문이다. 그래서 "30005번 포트는 안전하니까 열어줘!"라고 명령해야 한다.


# gcloud 명령으로 포트를 오픈한다. 
gcloud compute firewall-rules create allow-tcp-30005 \
  --direction=INGRESS \
  --rules=tcp:30005 \
  # ... (생략)

INGRESS: 들어오는 트래픽(Inbound)을 의미한다.
tcp:30005: 30005번 포트로 들어오는 TCP 신호를 허용하겠다는 뜻이다.

이 명령어를 치고 Creating firewall...done. 메시지가 떴다면 구글 클라우드 경비실에 통행증을 발급받은 셈이다.

  1. 결과 확인 및 문제 해결
    kubectl get svc 결과 분석

NAME                  TYPE           CLUSTER-IP       EXTERNAL-IP     PORT(S)
tomcat-loadbalancer   LoadBalancer   34.118.xxx.xxx   34.30.yyy.yyy   80:30006/TCP
tomcat-nodeport       NodePort       34.118.xxx.xxx   <none>          8080:30005/TCP

tomcat-loadbalancer: EXTERNAL-IP에 34.30.yyy.yyy이 할당되었다. 브라우저에 http://34.30.yyy.yyy 치면 톰캣이 보여야 정상이다. (성공!)

tomcat-nodeport: EXTERNAL-IP가 <none>이다. 왜냐하면 얘는 노드(컴퓨터) 자체의 IP를 쓰기 때문이다. [노드 IP]:30005로 접속해야 한다.

  • 마무리

코딩을 전혀 모르는 사람도, 머릿속 아이디어를 실제 동작하는 앱으로 만들 수 있는 시대가 왔습니다.

이 글에서는 claude.ai 웹 채팅에서 아이디어를 구체화하고, Claude Code에서 실제 개발까지 완료하는 전체 워크플로우를 소개합니다.

💡 이 가이드의 핵심 전략: "기획은 채팅으로, 개발은 에이전트로"

⑥ 결과 확인 및 피드백 반복

Step 1. claude.ai에서 아이디어 구체화하기

Claude Code를 바로 열고 "이거 만들어줘"라고 하면 십중팔구 원하는 결과가 나오지 않습니다. 먼저 웹 채팅에서 아이디어를 충분히 다듬는 것이 핵심입니다.

왜 채팅에서 먼저 해야 하나요?

  • Claude Code는 코딩 실행 환경이라 토큰 소모가 빠릅니다
  • 채팅에서는 부담 없이 여러 번 대화하며 방향을 잡을 수 있습니다
  • "뭘 만들지"가 명확해야 "어떻게 만들지"도 정확해집니다

대화 예시

이런 식으로 시작하면 됩니다:

회사에서 팀원들 커피 주문을 취합하는 웹앱을 만들고 싶어. 카카오톡으로 링크를 공유하면 각자 메뉴를 선택하고, 취합자가 한눈에 볼 수 있는 서비스야.

Claude가 이런 질문들을 던질 겁니다:

  • 사용자는 몇 명 정도인가요?
  • 로그인이 필요한가요?
  • 메뉴는 어떻게 관리하나요?
  • 모바일에서도 잘 보여야 하나요?

이 대화를 3~5번 정도 주고받으면, 아이디어가 꽤 구체적인 기획으로 바뀝니다.

💡 Tip: 더 좋은 결과를 위한 대화법

아이디어가 막연할수록 아래 내용을 함께 알려주면 좋습니다:

  • 누가 쓰나요? → "우리 팀 8~15명이 쓸 거야"
  • 어디서 쓰나요? → "모바일 위주, 카톡으로 공유"
  • 비슷한 서비스가 있나요? → "배민 장바구니 공유 같은 느낌"
  • 반드시 필요한 기능은? → "주문 마감 시간 설정"
  • 절대 필요 없는 것은? → "결제 기능은 필요 없어"

Step 2. 3대 문서 생성 요청

아이디어가 충분히 구체화되었다면, 같은 대화창에서 아래와 같이 요청합니다:

지금까지 대화한 내용을 바탕으로 PRD, TRD, TASKS 3개 문서를 만들어줘.

각 문서가 하는 역할

문서 역할 비유
PRD(Product Requirements Document, 제품 요구사항 문서) 무엇을 만들 것인가 건축의 "설계 의뢰서"
TRD(Technical Requirements Document, 기술 요구사항 문서) 어떤 기술로 만들 것인가 건축의 "구조 설계도"
Tasks(구현 작업 목록) 어떤 순서로 만들 것인가 건축의 "공정표"

📌 문서 품질을 높이는 추가 요청

기본 문서가 생성된 후, 아래 내용을 추가로 요청하면 결과물의 완성도가 올라갑니다:

"PRD에 아래 항목들이 포함되어 있는지 확인하고 빠진 부분을 보완해줘:
- 사용자 페르소나 (누가 쓰는지)
- 핵심 사용자 시나리오 (어떤 상황에서 어떻게 쓰는지)
- MVP 범위 (1차로 꼭 만들 기능 vs 나중에 추가할 기능)
- 성공 기준 (이게 잘 만들어졌다는 걸 어떻게 판단하는지)"

"TRD에 아래 내용도 포함시켜줘:
- 기술 스택 선택 이유
- 폴더 구조
- 외부 API나 서비스 의존성
- 배포 방법 (Vercel, Cloudflare Pages 등)"

Tasks는 에이전트가 바로 실행할 수 있도록 체크리스트 형태로 만들어줘. 각 태스크에 예상 소요 시간도 넣어줘.

⚠️ 문서 검토는 반드시 직접 하세요

Claude가 만든 문서를 그대로 넘기지 마세요. 꼭 한 번 읽어보고 확인할 것:

  • 빠진 기능이 없는지
  • 불필요한 기능이 들어가 있진 않은지 (과도한 기능은 복잡도를 높입니다)
  • 기술 스택이 내 상황에 맞는지 (예: 서버가 필요 없는데 백엔드가 포함되어 있진 않은지)
  • 태스크 순서가 논리적인지

수정이 필요하면 같은 대화에서 바로 피드백하면 됩니다:

"PRD에서 관리자 기능은 빼줘. MVP에는 필요 없어."
"TRD에서 DB는 Supabase 대신 SQLite로 바꿔줘. 서버 비용 없이 하고 싶어."

Step 3. 문서를 로컬 프로젝트 폴더에 복사

생성된 3개 파일을 다운로드한 뒤, 작업할 폴더에 넣습니다.

my-coffee-app/
├── PRD.md
├── TRD.md
├── Tasks.md
└── (아직 비어 있음 — Claude Code가 여기를 채울 겁니다)

💡 docs/ 폴더에 넣고 싶을 수 있지만, docs/는 나중에 MkDocs 같은 도구로 프로젝트 문서 사이트를 만들 때 사용하는 폴더입니다. 기획 문서는 프로젝트 루트에 두세요.

Claude Code 설치가 안 되어 있다면?

터미널(맥: Terminal, 윈도우: PowerShell)을 열고:

npm install -g @anthropic-ai/claude-code

Node.js가 설치되어 있어야 합니다. 없다면 nodejs.org에서 LTS 버전을 먼저 설치하세요.

설치 확인:

claude --version

Step 4. Claude Code에서 에이전트 팀으로 개발하기

에이전트 팀이란?

에이전트 팀은 여러 개의 Claude Code 세션이 동시에 협업하는 구조입니다. 하나의 세션이 팀 리드(Team Lead)가 되고, 나머지 팀메이트(Teammate)들이 각자 독립된 컨텍스트 윈도우에서 병렬로 작업합니다.

기존의 서브에이전트(Subagent)와 다른 점은, 팀메이트들이 서로 직접 메시지를 주고받을 수 있다는 것입니다. 서브에이전트는 메인 에이전트에게만 결과를 보고하지만, 에이전트 팀의 팀메이트들은 서로 발견한 내용을 공유하고 조율합니다.

┌─────────────────────────────────────┐
│           팀 리드 (Lead)             │
│    작업 배분, 조율, 결과 종합         │
│         ↕          ↕          ↕     │
│    팀메이트 A   팀메이트 B   팀메이트 C │
│    (프론트)     (백엔드)     (테스트)  │
│         ↔          ↔                │
│       서로 직접 메시지 교환 가능       │
└─────────────────────────────────────┘
     ▼ 공유 태스크 리스트 ▼
  [✅ 완료] [🔄 진행중] [⏳ 대기]

사전 준비: 에이전트 팀 활성화

에이전트 팀은 실험적 기능이라 기본적으로 꺼져 있습니다. 먼저 활성화해야 합니다:

# 환경변수로 활성화 (터미널에서 실행)
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

또는 ~/.claude/settings.json에 추가:

{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}

에이전트 팀 실행

프로젝트 폴더로 이동한 뒤 Claude Code를 실행합니다:

cd my-coffee-app
claude --dangerously-skip-permissions

💡 --dangerously-skip-permissions는 Claude Code가 파일 생성, 명령어 실행 등을 할 때마다 "허용할까요?" 묻지 않고 자동으로 진행하게 해줍니다. 이게 없으면 매번 수십 번씩 승인 버튼을 눌러야 해서 에이전트 팀이 제대로 돌아가지 않습니다.
>
⚠️ 주의: 이름에 "dangerously"가 들어간 이유가 있습니다. 중요한 파일이 있는 폴더에서는 사용하지 마세요. 반드시 새로 만든 빈 프로젝트 폴더에서만 사용하세요.

Claude Code가 열리면 아래와 같이 요청합니다:

"PRD.md, TRD.md, Tasks.md를 읽고, 이 프로젝트를 구현하기 위한 에이전트 팀을 구성해줘. 프론트엔드, 백엔드, 테스트 담당으로 팀메이트를 나눠서 병렬로 진행해줘."

Claude가 팀 구성안을 제안하면 확인(승인)해야 팀이 생성됩니다. Claude는 승인 없이 팀을 만들지 않습니다.

팀이 구성되면 일어나는 일

  1. 공유 태스크 리스트가 만들어집니다 — Tasks.md 기반으로 작업 목록이 생성되고, 의존성(순서)이 자동 관리됩니다
  2. 팀메이트들이 태스크를 자동 배정받습니다 — 하나를 끝내면 다음 대기 중인 태스크를 스스로 가져갑니다
  3. 팀메이트 간 메시지 교환이 이루어집니다 — 예를 들어 백엔드 팀메이트가 API를 완성하면 프론트엔드 팀메이트에게 알려줍니다

팀메이트 확인 및 직접 소통

터미널에서 Shift+↑/↓로 팀메이트를 선택하고, 직접 메시지를 보낼 수 있습니다. 팀 리드를 거치지 않고 특정 팀메이트에게 바로 피드백할 수 있습니다.

💡 에이전트 팀 vs 단일 세션, 언제 뭘 쓸까?

상황 추천
파일 하나 수정, 간단한 버그 픽스 단일 세션
여러 영역이 동시에 작업 가능한 프로젝트 에이전트 팀
빠른 조사나 검증이 필요한 부분 작업 서브에이전트

⚠️ 비용 참고: 에이전트 팀은 각 팀메이트가 독립된 Claude 세션이므로, 단일 세션보다 토큰을 많이 사용합니다. 간단한 작업에는 단일 세션이 더 효율적입니다.

Step 5. 개발 실행 요청

에이전트 팀이 구성되면, 개발을 시작합니다:

"Tasks.md에 정의된 작업 순서대로 개발을 시작해줘. 각 태스크가 완료될 때마다 Tasks.md의 체크리스트를 업데이트해줘."

개발 진행 중 알아두면 좋은 것들

1. 한 번에 다 되지 않습니다

Claude Code가 만든 코드에 에러가 날 수 있습니다. 당황하지 마세요. 에러 메시지를 그대로 붙여넣으면 Claude Code가 스스로 수정합니다:

"이런 에러가 나와: [에러 메시지]. 수정해줘."

2. 중간중간 확인하세요

태스크 3~4개가 완료될 때마다 실제로 실행해서 확인하세요:

# 웹앱이라면
npm run dev

브라우저에서 localhost:3000 (또는 안내된 주소)을 열어 직접 눌러보세요.

3. 피드백은 구체적으로

❌ "이거 좀 이상해"
✅ "메뉴 선택 화면에서 '확인' 버튼을 누르면 아무 반응이 없어. 주문 목록에 추가되어야 해."

❌ "디자인 좀 바꿔줘"
✅ "메인 화면의 배경색을 흰색으로, 버튼은 파란색(#3B82F6)으로 바꿔줘."

4. 세션이 길어지면 요약을 요청하세요

Claude Code 대화가 길어지면 컨텍스트가 흐려질 수 있습니다:

"지금까지 완료된 작업과 남은 작업을 정리해줘."

이렇게 하면 Claude Code가 현재 상태를 다시 정리하고, 이어서 진행할 수 있습니다.

Step 6. 결과 확인 및 배포

모든 태스크가 완료되면:

"모든 기능이 정상 동작하는지 최종 점검해줘. 그리고 Vercel(또는 Cloudflare Pages)에 배포하는 방법을 알려줘."

배포 후 체크리스트

  • [ ] 모바일에서 잘 보이는지
  • [ ] 링크 공유가 되는지
  • [ ] 주요 기능이 모두 동작하는지
  • [ ] 에러 발생 시 사용자에게 안내가 나오는지

자주 하는 실수 & 해결법

1. "아이디어 구체화 없이 바로 Claude Code에 던지기"

문제: "커피 주문 앱 만들어줘"만으로는 Claude Code가 수십 가지를 임의로 결정합니다. 원하는 것과 다른 결과물이 나올 확률이 높습니다.

해결: 반드시 채팅에서 먼저 구체화하세요. 10분의 대화가 2시간의 수정을 줄여줍니다.

2. "문서를 검토하지 않고 바로 개발 요청"

문제: PRD에 불필요한 기능(결제, 소셜 로그인 등)이 잔뜩 들어가면 개발이 복잡해지고 에러도 많아집니다.

해결: MVP(최소 기능 제품)에 집중하세요. "이게 없으면 서비스가 안 되는 기능"만 남기세요.

3. "에러가 나면 포기하기"

문제: 처음 실행할 때 에러가 나는 건 정상입니다.

해결: 에러 메시지를 복사해서 Claude Code에 붙여넣기만 하면 됩니다. 대부분 자동으로 고칩니다.

4. "한 세션에서 너무 많은 것을 요청하기"

문제: Claude Code의 컨텍스트 윈도우에는 한계가 있습니다. 대화가 너무 길어지면 앞부분을 잊습니다.

해결: 큰 프로젝트는 단계별로 나누세요. "Phase 1: 기본 기능" → "Phase 2: 디자인 개선" → "Phase 3: 추가 기능" 식으로 세션을 분리하는 것이 좋습니다.

5. "기술 스택을 Claude에게 완전히 맡기기"

문제: Claude가 제안한 기술 스택이 내 상황(서버 비용, 유지보수 난이도)에 안 맞을 수 있습니다.

해결: TRD 검토 시 아래를 확인하세요:
- 서버 비용이 발생하는지 (무료 티어로 가능한지)
- 외부 서비스 의존성이 많지 않은지
- 나중에 내가 (또는 Claude에게 시켜서) 수정할 수 있는 구조인지

실전 예시: 처음부터 끝까지

아래는 실제로 이 워크플로우를 적용한 예시입니다.

Phase 1: 아이디어 → 문서 (claude.ai 채팅, 약 15분)

나: 회사에서 점심 메뉴 투표 앱을 만들고 싶어. 매일 아침 투표를 열고, 
    점심시간 전에 마감되는 거야.

Claude: 좋은 아이디어네요! 몇 가지 확인할게요...
        (3~5회 대화)

나: 지금까지 내용으로 PRD, TRD, Tasks 문서 만들어줘.

Claude: (3개 문서 생성)

나: PRD에서 댓글 기능은 빼줘. Tasks에서 예상 시간도 넣어줘.

Claude: (수정된 문서 생성)

Phase 2: 개발 (Claude Code, 약 30~60분)

나: docs 폴더의 문서를 읽고 에이전트 팀을 구성해줘.
Claude Code: (에이전트 팀 구성 완료)

나: Tasks.md 순서대로 개발 시작해줘.
Claude Code: (개발 진행...)

나: npm run dev 했는데 투표 버튼이 안 눌려.
Claude Code: (수정 후 재실행)

나: 완료! 배포 방법 알려줘.
Claude Code: (배포 가이드 제공)

마무리

정리하면 이렇습니다:

  1. 채팅에서 충분히 대화하세요 — 아이디어가 구체적일수록 결과물이 좋습니다
  2. 3대 문서(PRD, TRD, Tasks)를 만드세요 — 이것이 에이전트의 설계도입니다
  3. 문서를 반드시 검토하세요 — 10분의 검토가 2시간의 수정을 줄입니다
  4. 에이전트 팀으로 개발하세요 — 역할 분담이 품질을 높입니다
  5. 피드백은 구체적으로 하세요 — 스크린샷, 에러 메시지, 구체적 위치를 알려주세요

코딩을 몰라도 됩니다. 무엇을 만들고 싶은지 설명할 수 있으면 충분합니다.

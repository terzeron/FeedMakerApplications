FFmpeg의 내부를 들여다보다: 라이브러리 API로 직접 만드는 미디어 플레이어

@jaehong
2026년 3월 22일 3:55 오전
Tech

개발자라면 누구나 ffmpeg -i input.mp4 output.avi 정도는 써봤을 것이다. 하지만 FFmpeg을 CLI 도구로만 아는 것과, 그 안의 라이브러리 구조를 이해하고 자기 애플리케이션에 직접 통합할 수 있는 것은 완전히 다른 레벨의 역량이다. Igalia의 엔지니어 Loïc Le Page가 공개한 이 튜토리얼은 FFmpeg의 C 라이브러리 API를 사용해 멀티미디어 파일을 열고, 스트림을 분리(demux)하고, 디코딩하는 전체 과정을 코드 레벨에서 차근차근 보여준다. HN 커뮤니티에서 "FFmpeg is a superpower"라는 반응이 나올 만큼, 이 도구의 내부 동작 원리를 아는 것은 미디어 처리가 필요한 모든 프로젝트의 기반이 된다.

CLI 뒤에 숨은 진짜 FFmpeg: 8개의 라이브러리

대부분의 개발자가 FFmpeg이라고 하면 커맨드라인 도구를 떠올린다. ffmpeg(포맷 변환), ffplay(간이 플레이어), ffprobe(스트림 분석기). 하지만 이 도구들은 사실 빙산의 일각이다. FFmpeg의 핵심은 그 아래에 있는 8개의 C 라이브러리이며, CLI 도구들도 결국 이 라이브러리 위에서 돌아간다.

구조를 한번 정리해보면 이렇다. libavformat이 컨테이너 포맷(MP4, MKV, AVI 등)을 열고 스트림을 분리하는 I/O 계층을 담당한다. libavcodec이 H.264, AAC 같은 코덱으로 인코딩/디코딩을 수행한다. libavfilter는 영상에 필터를 적용하는 그래프 기반 처리 엔진이고, libswscale과 libswresample은 각각 이미지 스케일링/색상 변환과 오디오 리샘플링을 처리한다. libavutil은 이 모든 라이브러리가 공유하는 유틸리티, libavdevice는 카메라나 마이크 같은 하드웨어 입출력, libpostproc은 디블로킹 같은 후처리 필터를 제공한다.

이 구조가 왜 중요하냐면, 자기 애플리케이션에 미디어 처리 기능을 넣을 때 FFmpeg CLI를 subprocess로 호출하는 것과 라이브러리를 직접 링크하는 것 사이에는 성능, 제어 가능성, 에러 핸들링 모든 면에서 큰 차이가 있기 때문이다. 프로토타입이나 스크립트 레벨에서는 CLI로 충분하지만, 프로덕션 레벨의 미디어 서비스(트랜스코딩 파이프라인, 실시간 스트리밍, 편집 도구 등)를 만들려면 라이브러리 API를 직접 다루는 것이 사실상 필수다.

5개의 핵심 구조체로 읽는 디코딩 파이프라인

FFmpeg 라이브러리 API의 핵심은 다섯 가지 C 구조체를 이해하는 것에서 시작한다. 이 튜토리얼이 명쾌한 점은 이 구조체들의 역할과 관계를 실제 코드 흐름 속에서 자연스럽게 보여준다는 것이다.

AVFormatContext는 가장 상위 레벨의 컨테이너다. 멀티미디어 파일을 열면 이 구조체가 파일 전체의 메타데이터, 스트림 정보, 동기화 정보를 관리한다. MP4 파일 하나를 열면 그 안에 비디오 스트림, 오디오 스트림, 자막 스트림 등이 들어있는데, 각각이 AVStream으로 표현된다. 각 스트림은 자신이 어떤 코덱으로 인코딩되어 있는지 알고 있고, 이 정보를 통해 AVCodec(디코더)을 찾아서 AVCodecContext(디코딩 컨텍스트)를 만든다.

실제 디코딩 과정은 이런 흐름이다. avformat_open_input()으로 파일을 열고 avformat_find_stream_info()로 스트림을 분석한 뒤, av_read_frame()으로 인코딩된 패킷(AVPacket)을 하나씩 꺼낸다. 이 패킷을 avcodec_send_packet()으로 디코더에 밀어 넣으면, avcodec_receive_frame()으로 디코딩된 로우 프레임(AVFrame)을 꺼낼 수 있다.

여기서 설계적으로 눈여겨볼 부분이 있다. 디코더가 FIFO 큐처럼 동작한다는 점이다. 패킷을 보내는 것(send)과 프레임을 받는 것(receive)이 분리되어 있고, 하나의 패킷이 반드시 하나의 프레임에 대응하지 않는다. 코덱 구현에 따라 내부적으로 멀티스레드 디코딩을 할 수도 있고, B-프레임 같은 참조 프레임 때문에 여러 패킷을 받아야 하나의 프레임이 나올 수도 있다. 이 비동기적 send/receive 패턴은 FFmpeg API를 처음 접하는 개발자가 가장 헷갈려하는 부분인데, 실제로 AVERROR(EAGAIN) 반환값을 제대로 처리하지 않아서 생기는 버그가 많다.

빌드 시스템과 접근성

이 튜토리얼의 또 다른 장점은 빌드 환경 설정을 극도로 간소화했다는 것이다. Meson 빌드 시스템을 사용해서 meson setup build만 실행하면 시스템에 FFmpeg이 없을 경우 자동으로 다운로드까지 해준다. FFmpeg 라이브러리 개발의 진입 장벽 중 하나가 빌드 환경 설정의 복잡함인데, 이 부분을 깔끔하게 해결한 셈이다.

다만 이 튜토리얼은 의도적으로 "101" 수준에 머무른다. 디먹싱과 디코딩의 기본 흐름만 다루고, 인코딩, 필터 체인, 하드웨어 가속(VA-API, NVDEC 등), 네트워크 스트리밍 같은 실전에서 필요한 주제는 다루지 않는다. 실제 프로덕션에서 FFmpeg 라이브러리를 쓰려면 메모리 관리(av_packet_unref, av_frame_free 같은 리소스 정리), 에러 핸들링, 스레드 안전성 같은 문제를 훨씬 깊이 파고들어야 한다.

커뮤니티가 추천하는 다음 단계

HN 토론에서 여러 개발자가 더 깊이 파고들고 싶은 사람들을 위한 리소스를 공유했다. 특히 Leandro Moreira의 ffmpeg-libav-tutorial이 가장 많은 추천을 받았는데, FFmpeg과 libav의 내부 동작을 가장 포괄적으로 설명하는 자료로 평가받고 있다. 트랜스코딩, 트랜스먹싱, 필터링까지 단계별로 다루기 때문에 이번 튜토리얼로 기본을 잡은 뒤 다음 단계로 넘어가기에 적합하다.

"FFmpeg is a superpower"라고 표현한 한 개발자의 말처럼, FFmpeg은 단순한 포맷 변환 도구가 아니라 사실상 모든 미디어 처리의 기반 인프라다. 브라우저의 비디오 재생, OTT 서비스의 트랜스코딩 파이프라인, 영상 편집 소프트웨어, 실시간 스트리밍 플랫폼까지, 미디어를 다루는 거의 모든 소프트웨어가 직간접적으로 FFmpeg에 의존한다. CLI로만 쓰던 FFmpeg의 라이브러리 레이어를 이해하는 것은, 미디어 기술의 블랙박스를 열어보는 첫 번째 열쇠가 된다.

프로젝트 없는 .NET, 새로운 플랫폼의 서막

@devwrite
2025년 9월 22일 3:00 오후
Shebang C# 바이브코딩 Filebasedapp 닷넷 스크립트 .net Fba

닷넷 개발은 오랫동안 Visual Studio, Rider 같은 풀스택 IDE 중심으로 굴러왔습니다. 강력하지만 무겁고, 학습 진입 장벽이 높다는 인식을 벗어나기 어려웠습니다.

그런데 .NET 10 Preview에서 등장한 File-based Applications (FBA) 기능은 이런 구조에 균열을 냅니다. 이제는 단순한 app.cs 파일 하나만으로도 다음처럼 실행할 수 있습니다.

dotnet run app.cs

프로젝트 파일도, 솔루션도 필요 없습니다. 코드만으로 시작할 수 있는 길이 열린 것입니다.

FBA, 단순함을 넘어 새로운 개발 경험

FBA는 단순히 “프로젝트 없이 코드를 실행하는 기능”에 그치지 않습니다.

파일 내부에 지시어(directive)를 적어두면, 패키지 참조부터 SDK 지정, 빌드 속성까지 한 번에 정의할 수 있습니다.

#!/usr/bin/env dotnet
#:package Spectre.Console@0.48.0
#:property OutputType=Exe

using Spectre.Console;

AnsiConsole.MarkupLine("[green]Hello, FBA![/]");

FBA 전용 지시자는 기존의 SDK 스타일 C# 프로젝트에서 사용하던 일부 설정을 C# 코드 레벨에서 지정할 수 있도록 기능을 보강한 것으로, 예전에 보셨던 C# 스크립트와는 다른 유형의 지시자 코드입니다. 거기다, 위의 코드에서 보듯 Shebang 라인도 FBA 문법의 일부로 취급되어 Bash 셸에서 바로 실행할 수도 있습니다.

이런 방식을 통해 FBA는 닷넷을 스크립트 언어처럼 가볍게 만들면서도, 기존 빌드 시스템과의 확장성까지 확보합니다. 즉, 흔한 “학습용 장난감”이 아니라, 플랫폼 자체로서 닷넷을 경량화하는 접근입니다.

IDE 시대를 넘어, CLI 중심 개발로

닷넷은 그동안 사상의 대부분이 IDE 종속적이었습니다.

Visual Studio는 Windows에 묶여 있었고, Rider는 뛰어나지만 상업용 라이선스가 필요하기에 IDE에 투자할 예산을 확보할 수 있는 조직이 아니라면 편하게 사용하기에 역시 어려움이 있습니다. (Visual Studio for Mac은 단종되었으며, 이 제품을 대체하기 위해 나온 C# Dev Kit은 Visual Studio Community Edition과 동일 라이선스를 요구하므로 엄밀하게 말해서 무료 제품이 아닙니다.)

이런 상황이다보니 작은 툴이나 단발성 프로젝트를 만들려는 개발자들에게 과도한 부담이었습니다.

FBA는 이 벽을 허물 수 있습니다.

  • 에디터 + Roslyn LSP Server + CLI만으로도 충분히 닷넷 개발 가능
  • Python이나 Node.js처럼, 가볍게 시작하고 필요하면 확장하는 흐름 지원
  • CI/CD 파이프라인, DevOps 자동화 도구, 셸 스크립트 대체재로 닷넷을 활용할 가능성 확대

바이브코딩(Vibe-coding)과 FBA

이번 변화는 바이브코딩(Vibe-coding)에도 유리합니다.

  • IDE 환경에서 바이브코딩은 설정, 프로젝트 생성, 패키지 관리 같은 단계가 필연적으로 끼어들었습니다. 즉, “즉흥적으로 코드를 치면서 아이디어를 검증”하는 흐름이 자주 끊겼습니다.
  • 반면 FBA는 dotnet run app.cs 한 줄로 바로 실행되기 때문에, 코드 작성과 실행 사이의 장벽이 거의 사라집니다.
  • NuGet 패키지도 파일 안에서 바로 선언 가능하니, 필요한 라이브러리를 호출하듯 즉석에서 불러올 수 있습니다.

이 덕분에 FBA는 단순히 “학습자용 기능”이 아니라, 코딩의 몰입감을 해치지 않는 실험 환경을 제공합니다. Python REPL이나 Node.js의 빠른 프로토타이핑 경험을 닷넷에서도 구현할 수 있는 셈입니다.

아직은 프리뷰

.NET 10이 2025년 11월에 출시됨에 따라 FBA도 기능이 같이 포함됩니다. 그러나 FBA 자체는 Preview 단계입니다.

  • 여러 파일과 프로그래밍 언어를 지원하는 것은 아직 제한적이고,
  • IDE 통합, 디버깅, 퍼블리시 기능도 완전하지 않습니다.

하지만 이는 출발점일 뿐입니다. 앞으로 Native AOT, 멀티파일 지원, 배포 단위 최적화가 이어진다면, FBA는 단순한 편의 기능을 넘어 실무에서 쓰이는 새로운 타입의 애플리케이션 플랫폼으로 자리잡을 것입니다.

결론: 개발자에게 열린 새로운 문

이번 업데이트는 단순한 기능 추가가 아닙니다.

  • IDE 종속적 생태계에서의 해방
  • 스크립트 친화적 닷넷 플랫폼
  • 바이브코딩에 최적화된 새로운 개발 경험

dotnet run app.cs는 작지만 의미 있는 전환점입니다.

이 기능을 단순히 실험적 옵션으로 치부하기보다, 닷넷 생태계의 체질 변화를 알리는 신호로 보는 것이 옳습니다. 지금은 Preview이지만, 개발자라면 지금부터 이 가능성을 시험해보고 체감해볼 필요가 있습니다.

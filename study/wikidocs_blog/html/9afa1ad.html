🗺️ [2회] 반도체 설계 흐름 한눈에 보기

— RTL부터 테이프아웃까지, 칩은 어떻게 만들어지는가

스마트폰 AP 하나가 세상에 나오기까지, 수백 명의 엔지니어가 수년을 함께 달립니다.
그런데 그 과정은 도대체 어떤 순서로 진행되는 것일까요?

소프트웨어 개발에는 "기획 → 개발 → 테스트 → 배포"라는 흐름이 있듯이, 반도체 설계에도 정해진 흐름이 있습니다.
RTL 설계 → 검증 → 합성 → 배치·배선 → 테이프아웃, 이 다섯 단계입니다.
오늘은 각 단계가 현업에서 어떤 의미를 갖는지, 큰 그림부터 잡아보겠습니다.

전체 흐름 한눈에 보기

아래 흐름도를 먼저 살펴보세요.

RTL 설계 — "코드로 회로를 그린다"

설계의 출발점은 RTL(Register Transfer Level) 코드 작성입니다.
설계 엔지니어는 Verilog나 SystemVerilog라는 하드웨어 기술 언어로 회로의 동작을 기술합니다.
소프트웨어 개발로 치면 소스 코드를 작성하는 단계와 비슷하지만, 결정적인 차이가 있습니다.
소프트웨어 코드는 CPU 위에서 순서대로 실행되지만, RTL 코드는 실리콘 위에 실제로 만들어질 회로의 동작을 기술하는 것입니다.

현업의 RTL 엔지니어는 단순히 코드를 잘 짜는 것만으로는 부족합니다.
전력·성능·면적, 세 가지 상충 관계(PPA, Power-Performance-Area)를 항상 머릿속에 담고 설계해야 합니다.

기능 검증 — "만들기 전에 부수어 본다"

RTL 코드가 어느 정도 완성되면 곧바로 검증이 시작됩니다.
검증 엔지니어는 설계가 의도한 대로 동작하는지 시뮬레이션으로 철저히 확인합니다.
이 단계가 존재하는 이유는 단순합니다.
반도체는 한번 제조하면 수정이 불가능하기 때문입니다.

업계에서는 흔히 "설계 1명이 버그를 만들면, 검증 3명이 찾는다"고 말합니다. 실제로 대형 SoC 프로젝트에서 검증 팀이 전체 인원의 60~70%를 차지하는 경우도 드물지 않습니다.

이 연재에서 검증(UVM/시뮬레이션)에 가장 많은 분량을 할애하는 것도 바로 이 이유입니다.

논리 합성 — "코드를 게이트로 변환한다"

검증을 통과한 RTL 코드는 합성 도구(Synthesis Tool)에 의해 AND, OR, NAND 같은 실제 논리 게이트들의 연결망, 즉 넷리스트(Netlist)로 자동 변환됩니다.
이 작업은 Synopsys의 Design Compiler 같은 EDA 소프트웨어가 처리합니다.

중요한 점은, 합성 도구가 단순한 변환에 그치지 않는다는 것입니다.
엔지니어가 "이 회로는 1GHz로 동작해야 한다"는 제약 조건을 전달하면, 도구는 그에 맞게 회로를 최적화하기도 합니다.

배치·배선(P&R) — "실리콘 위에 자리를 잡다"

넷리스트를 받아 각 게이트를 실리콘 칩의 물리 공간 위에 배치(Place)하고 금속 선으로 연결(Route)하는 작업입니다.
아파트 평면도를 설계하고 전선 배관을 배치하는 과정에 비유할 수 있습니다.

수십억 개의 트랜지스터를 손톱만 한 면적 안에 욱여넣어야 하므로, 고성능 서버에서 수 시간에서 수십 시간이 걸리는 연산이 필요한 경우도 많습니다.

테이프아웃 — "이제 공장의 차례"

모든 검증을 통과하면 설계 데이터(GDS 파일)를 TSMC, 삼성 파운드리 같은 반도체 제조사에 넘깁니다.
이것이 테이프아웃(Tapeout) 입니다.
이름의 유래는 과거에 설계 데이터를 자기 테이프에 담아 공장으로 보냈던 시절에서 비롯되었습니다.

테이프아웃 이후에는 설계 팀이 할 수 있는 일이 없습니다.
수개월 뒤 웨이퍼가 돌아올 때까지 기다릴 뿐입니다.
그래서 테이프아웃 직전은 팀 전체가 가장 긴장하는 시간이기도 합니다.

다음 편 예고

다음 회차에서는 이 설계 흐름이 실제로 작동하는 무대, SoC 내부를 해부해봅니다.
CPU, GPU, NPU, 메모리 컨트롤러가 하나의 칩 안에서 어떻게 역할을 나누고 협력하는지 살펴볼 예정입니다.

💬 한 줄 정리 : 반도체 설계는 RTL 코드 작성 → 시뮬레이션 검증 → 합성 → 배치·배선 → 테이프아웃 순으로 진행되며, 한번 공장에 보내면 수정이 불가능하기 때문에 각 단계의 검증이 절대적으로 중요하다.

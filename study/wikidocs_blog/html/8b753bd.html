Deno의 대규모 감원, 그리고 사라진 CEO: 오픈소스 런타임의 비즈니스 모델은 가능한가

@jaehong
2026년 3월 22일 3:57 오전
Tech

지난주 Deno Land Inc.의 직원 절반 가까이가 회사를 떠났다. 레이오프다. 그런데 더 이상한 일이 벌어졌다. deno.com에 접속하면 404 에러가 떴고, CEO Ryan Dahl은 아무런 공식 입장도 내놓지 않았다. Node.js를 만든 전설적인 엔지니어가 "Node의 실수를 바로잡겠다"며 시작한 프로젝트가 5년 만에 맞닥뜨린 현실이다. 시드 490만 달러, 시리즈 A 2,100만 달러. 총 2,590만 달러의 벤처 자금을 태운 뒤 남은 건 무엇인가.

이 사건은 단순한 한 스타트업의 실패담이 아니다. "더 나은 오픈소스 도구를 만들면 개발자가 모이고, 개발자가 모이면 돈이 된다"는 DevTools 비즈니스의 근본 가설을 시험대에 올린다.

1년 전의 경고, 그리고 무시된 신호들

사실 조짐은 있었다. 약 1년 전 원문 저자가 "Deno의 쇠퇴"라는 글을 썼을 때, Deno Land Inc.의 상황은 이미 좋지 않았다. 그 글이 Hacker News 1위에 오르자 Ryan Dahl이 직접 공식 블로그에 반박 글을 올렸다. "Deno의 사망 보고는 크게 과장되었다"는 제목이었다.

Dahl은 Deno 2.0 출시 이후 월간 활성 사용자가 두 배로 늘었다고 주장했다. 하지만 구체적인 수치는 공개하지 않았다. "사용자가 두 배"라는 말은, 기저가 1만 명이면 2만 명이고 100만 명이면 200만 명이다. 투자사 Sequoia Capital이 기대한 성장 곡선이 어떤 것이었는지는 모르지만, 구체적 숫자 없는 성장 자랑은 보통 좋은 신호가 아니다.

Deno Deploy: 기술은 좋았지만 아무도 오지 않았다

Deno의 주된 수익원이었던 Deno Deploy는 흥미로운 제품이었다. V8 아이솔레이트(격리된 경량 실행 환경) 기반으로 서버리스 엣지 컴퓨팅을 제공했다. 문제는 실행 품질이었다. 아이솔레이트 시작 시간이 심하게 들쭉날쭉했다. 사용자들이 피드백을 보내도 묵묵부답이었다.

상황이 바뀐 건 웹 개발 인플루언서 Wes Bos가 이슈를 올린 뒤였다. 수십만 팔로워를 가진 유명 개발자의 목소리에야 비로소 반응한 것이다. Deno는 2025년 말 Deploy를 재출시하고, 올해 2월 "정식 출시(GA)"를 선언했다. 하지만 원문 저자의 물음은 날카롭다. "누가 쓰고 있나? 누가 신경이나 쓰나?"

JSR의 실패와 "두 번째 시스템 증후군"

Deno가 npm을 대체하겠다며 만든 JSR(JavaScript Registry)도 비슷한 궤적을 그렸다. JSR의 YouTube 채널은 거의 유령 도시 수준이고, GitHub 스타 성장 곡선을 보면 비슷한 시기에 등장한 NPMX에 크게 뒤처진다.

차이는 명확하다. NPMX는 기존 npm 생태계 위에서 "더 빠르고 편한 npm"을 지향했다. 반면 JSR은 개발자에게 새로운 패키지 생태계로의 이주를 요구했다. HN 토론에서 한 개발자가 정확히 짚었다. "개발자들은 Node와 npm을 대체하고 싶은 게 아니다. 지금 쓰는 걸 마찰 없이 더 좋게 만들어주길 원한다."

Deno가 초기에 고집했던 HTTP 임포트(npm 대신 URL로 패키지를 직접 가져오는 방식)도 마찬가지다. 이론적으로는 깔끔했지만 현실에선 버전 관리, 캐싱, 보안 문제가 산적했다. 결국 Deno 팀도 이를 인정하고 package.json 지원으로 방향을 틀었지만, 전환 과정에서 패키징 시스템이 혼란스러워지면서 오히려 상황을 악화시켰다.

HN 토론에서 Meteor(2012~2016년 웹 프레임워크 스타트업)의 첫 번째 엔지니어 출신 개발자가 남긴 분석이 인상적이다. "Deno는 런타임, 프레임워크, 린터, 문서, 호스팅, 패키징을 전부 한꺼번에 재발명하려 했다. 거의 모든 구성 요소가 기존 도구보다 열등했다. TypeScript 퍼스트클래스 지원과 보안 강화가 목표였다면, 고전적인 'embrace and extend' 전략을 썼어야 했다." 이건 Frederick Brooks가 말한 "두 번째 시스템 증후군"의 교과서적 사례다. 첫 번째 시스템(Node.js)의 문제를 너무 잘 아는 사람이, 두 번째 시스템에서 모든 걸 한꺼번에 고치려다 오히려 복잡성의 늪에 빠지는 것이다.

오픈소스 DevTools 기업의 딜레마

이 사건이 제기하는 더 큰 질문이 있다. 오픈소스 개발자 도구 회사의 지속 가능한 비즈니스 모델은 존재하는가?

HN 토론에서도 이 점이 핵심 쟁점으로 떠올랐다. 한 개발자는 "오픈소스 DevTools 회사의 지속가능한 모델이 있다고 확신하지 못하겠다. 보통 호스팅이나 코드 품질 도구로 수익화하겠다는 애매한 계획만 있는데, 호스팅은 어려운 사업이고 AWS라는 800파운드 고릴라가 버티고 있다"고 지적했다.

Docker, HashiCorp, Elastic, Redis, MongoDB. 오픈소스 인프라 도구 기업들의 수익화 여정은 하나같이 험난했다. 그나마 성공한 기업들은 엔터프라이즈 고객을 위한 관리형 서비스나 프리미엄 기능으로 차별화했다. Deno Deploy는 이 경로를 시도했지만, 전제 조건인 "충분한 사용자 기반"을 확보하지 못했다.

Bun이라는 변수, 그리고 경쟁의 역학

Deno의 하락세를 가속화한 건 2023년 등장한 Bun이었다. Zig로 작성된 Bun은 극단적인 성능을 전면에 내세우며 빠르게 관심을 끌었다. HN 댓글 중 한 개발자의 말이 현실을 잘 보여준다. "Bun의 극적인 등장, 그리고 그에 대한 Deno 측의 반응이 내 관심권에 전혀 들어오지 않은 것. 그게 사실상 런타임 전환에 대한 흥미를 증발시켰다."

흥미로운 점은, 원문 저자조차 Bun에 대해서도 "아무도 인정하지 않으려는 버그와 호환성 문제가 상당하다"고 꼬집는다는 것이다. Node는 여전히 TypeScript와 ESM 지원에서 마찰이 있고, Bun은 안정성 문제가 있으며, Deno는 생태계 호환성에서 해맸다. JavaScript 런타임 시장은 완벽한 승자가 없는 상태다.

AI 피벗이라는 마지막 카드

Ryan Dahl의 최근 트윗과 블로그 글을 보면 AI 쪽으로의 피벗을 암시하고 있다. OpenAI 인수설까지 돌았지만, 이는 근거 없는 추측에 가깝다. 다만 Deno 런타임의 보안 샌드박스(기본적으로 파일시스템, 네트워크 접근을 차단하는 권한 모델)가 AI 에이전트 실행 환경으로서 가치가 있을 수 있다는 시각은 존재한다.

그러나 AI 피벗은 이미 수많은 스타트업이 시도하고 있는 포화 전략이다. Deno 팀이 절반으로 줄어든 상태에서, 기존 사업도 안정시키지 못한 조직이 완전히 새로운 영역에서 경쟁력을 가질 수 있을지는 회의적이다.

그래서, Deno의 유산은 무엇인가

HN 토론에서 가장 공감을 얻은 의견들은 오히려 Dahl에 대한 존중과 Deno의 기여를 인정하는 목소리였다. "Deno는 생태계를 개선하는 힘이었다. TypeScript 퍼스트클래스 지원, 보안 모델, Web API 준수. 이런 아이디어들이 Node와 Bun을 더 좋게 만드는 압력이 되었다." Node.js 동료였던 Mikeal Rogers의 사망 소식까지 겹쳐 Dahl 개인에게도 힘든 시기라는 점을 언급하며, "좀 봐줘라"는 반응도 있었다.

냉정하게 보면, Deno의 기술적 비전은 옳았다. TypeScript 네이티브 지원, 보안 기본값, Web 표준 API 채택. 이건 JavaScript 런타임이 가야 할 방향이었고, 실제로 Node.js도 이 방향으로 움직이고 있다. 문제는 비전을 실행으로 옮기는 과정에서 너무 많은 것을 동시에 바꾸려 했고, 그 과정에서 기존 생태계와의 호환성이라는 가장 중요한 가치를 경시했다는 점이다.

Deno 런타임 자체는 오픈소스로 살아남을 수 있다. 하지만 Deno Land Inc.라는 벤처 기업의 미래는 CEO의 다음 한마디에 달려 있다. 레이오프 후 금요일이 지나도록 아무 말이 없다는 것. 그게 어쩌면 모든 답을 담고 있는지도 모른다.

Hypura: 맥 메모리보다 큰 LLM을 돌리는 스토리지 계층 인식 스케줄러

@jaehong
2026년 3월 26일 8:18 오전
Ai

32GB 맥에서 40GB 모델을 로드하면 어떻게 될까? 일반적인 llama.cpp라면 시스템이 스왑 지옥에 빠지다가 결국 프로세스가 죽는다. Hypura는 이 문제를 정면으로 해결한다. GPU, RAM, NVMe 세 계층에 모델 텐서를 지능적으로 배치해서, 물리 메모리를 초과하는 모델도 크래시 없이 추론을 돌린다. HN 커뮤니티에서는 이 접근법의 실용성을 둘러싸고 NVMe 접근 패턴, MoE 희소성의 실제 효과, 그리고 "느려도 돌아가는 것"의 가치에 대한 기술적 토론이 벌어졌다.

핵심 아이디어: 모든 텐서가 평등하지 않다

Hypura의 출발점은 단순하다. LLM의 모든 가중치가 동일한 빈도로 접근되지 않는다는 사실이다. 노름(norm)과 임베딩은 매 토큰마다 참조되지만 크기가 작다. 어텐션 레이어도 상대적으로 자주 쓰인다. 반면 FFN(Feed-Forward Network) 가중치는 전체 모델의 60% 이상을 차지하지만, MoE(Mixture of Experts, 여러 전문가 네트워크 중 일부만 활성화하는 구조) 모델이라면 매 토큰당 8개 중 2개만 실제로 동작한다.

Hypura는 GGUF 파일을 읽고, 하드웨어를 프로파일링한 뒤(GPU 워킹셋, RAM 대역폭, NVMe 순차 읽기 속도), 각 텐서를 GPU/RAM/NVMe 중 최적의 계층에 배치하는 최적화 문제를 푼다. 이 과정이 완전 자동이라는 점이 중요하다. 풀 버퍼 크기, 프리페치 깊이, 메모리 예산을 하드웨어 프로파일에서 자동 계산한다.

세 가지 추론 모드

하드웨어 조건과 모델 아키텍처에 따라 세 가지 모드가 자동으로 선택된다.

Full-resident 모드는 모델이 GPU+RAM에 온전히 올라갈 때 동작한다. NVMe I/O가 전혀 없으므로 Metal GPU 속도 그대로 나온다. 벤치마크에서 Qwen 2.5 14B(8.4GB)가 21 tok/s로, 일반 llama.cpp와 동일한 성능을 보여주면서 오버헤드가 제로임을 확인했다.

Expert-streaming 모드가 Hypura의 진짜 무기다. Mixtral 같은 MoE 모델에서 비전문가 텐서(약 1GB)만 GPU에 상주시키고, 전문가 텐서는 NVMe에서 필요할 때 스트리밍한다. 핵심은 라우터 인터셉션이다. 추론 콜백에서 어떤 전문가가 선택됐는지 실시간으로 파악하고, 선택된 전문가의 스트라이드만 NVMe에서 로드한다. 8개 중 2개만 활성화되니 I/O가 75% 줄어든다. 여기에 뉴런 캐시가 토큰 간 시간적 지역성(temporal locality)을 활용해 99.5%의 히트율을 달성한다. 공동 활성화 추적(co-activation tracking)은 다음에 어떤 전문가가 발동할지 예측해서 투기적 프리페치까지 수행한다. 결과적으로 31GB Mixtral 8x7B가 32GB 맥 미니에서 2.2 tok/s로 돌아간다.

Dense FFN-streaming 모드는 MoE가 아닌 밀집 모델용이다. 어텐션과 노름(약 8GB)을 GPU에 고정하고, FFN 가중치(약 32GB)를 NVMe에서 동적 풀 버퍼를 통해 스트리밍한다. 40GB Llama 70B가 0.3 tok/s로 동작한다. 느리지만, llama.cpp에서는 아예 OOM으로 실행 자체가 불가능한 모델이다.

NVMe 접근 패턴: 순차 vs 랜덤의 현실

HN 토론에서 가장 날카로운 기술적 지적이 나온 부분이다. 한 엔지니어가 핵심을 짚었다. "실제 읽기 패턴이 순차적이어서 NVMe 대역폭을 포화시킬 수 있는지, 아니면 어텐션 레이어 접근 패턴이 결국 랜덤에 가까워서 처리량을 죽이는지가 관건"이라는 것이다. 수치로 보면 격차가 크다. 일반적인 NVMe의 순차 읽기가 5~7 GB/s인 반면, 랜덤 읽기는 큐 뎁스에 따라 500 MB/s 수준까지 떨어진다.

이 지적에 대한 반론도 있었다. MoE의 희소 활성화가 핵심인데, 전체 2TB를 매번 읽는 게 아니라 활성화된 전문가만 읽기 때문에 총 I/O 양 자체가 크게 줄어든다는 점이다. 다만 여기에도 함정이 있다. MoE의 접근 패턴은 활성화될 전문가에 따라 결정되므로 본질적으로 랜덤에 가까워지고, 이는 NVMe의 최악 시나리오에 해당한다. Hypura의 공동 활성화 추적과 투기적 프리페치가 이 문제를 완화하려는 시도이지만, 모델 규모가 커질수록 이 전략의 효과가 유지될지는 미지수다.

1T 파라미터 모델을 예로 들면, FP16 기준 포워드 패스당 약 2TB의 가중치를 스트리밍해야 한다. 최대 순차 읽기 속도에서도 토큰 하나에 300초 이상이 걸린다. 대화형 사용은 사실상 불가능하고, 배치 추론이나 레이턴시에 무관한 워크로드에서만 의미가 있다.

"느려도 돌아가는 것"의 가치

0.3 tok/s가 과연 쓸모 있을까? HN의 한 댓글이 정곡을 찔렀다. "많은 로컬 워크로드에서 1 tok/s 미만은 포그라운드에서는 쓸모없지만, 백그라운드에서는 완전히 수용 가능하다. '크래시'와 '하룻밤에 완료'의 차이라면, 그건 의미 있는 능력의 도약이다."

이건 꽤 중요한 관점 전환이다. 지금까지 로컬 LLM 추론은 "실시간 대화"라는 프레임에 갇혀 있었다. 하지만 에이전트 기반 워크로드가 늘면서, 백그라운드에서 밤새 돌려도 되는 작업이 점점 많아지고 있다. 대량 문서 요약, 코드 리뷰, 데이터 분석 같은 배치 작업이라면 토큰 속도보다 "돌아가느냐 안 돌아가느냐"가 결정적이다.

OS 스왑으로 충분하지 않은가?

애플 실리콘의 유니파이드 메모리와 macOS의 강력한 하드웨어 통합을 고려하면, "그냥 스왑이 알아서 처리하면 안 되나?"라는 질문이 자연스럽다. 실제로 HN에서도 이 의문이 제기됐다. 답은 간단하다. OS 레벨 스왑은 모델 아키텍처를 모른다. 어떤 텐서가 매 토큰마다 필요하고 어떤 텐서가 드물게 접근되는지 알 길이 없으므로, 스왑 대상을 무작위로 선택하다가 스래싱에 빠진다. Hypura는 모델 구조를 이해하기 때문에 "이 노름 텐서는 절대 내보내지 마" 같은 판단을 할 수 있다.

참고로 한 댓글에서 기술적 정정도 있었다. macOS에는 리눅스 같은 전통적인 OOM 킬러가 없다. "스왑 공간 부족 킬러"가 있긴 하지만 상당히 수동적이어서, 실제로는 메모리 와이어링 실패가 먼저 발생하거나 시스템이 극도로 느려진 채 패닉 상태에 빠지는 게 더 흔한 시나리오다.

SSD 수명 걱정은 불필요하다

NVMe에서 모델 가중치를 스트리밍한다고 하면 SSD 수명을 걱정하는 건 자연스럽다. 하지만 Hypura의 NVMe I/O 경로는 전부 읽기 전용이다. F_NOCACHE 플래그와 pread() 시스템 콜로 GGUF 파일에서 텐서 가중치를 RAM/GPU 메모리 풀로 읽어올 뿐이다. SSD 마모는 쓰기 사이클(NAND 플래시의 프로그램/소거 사이클)에 의해 발생하므로, 읽기만 수행하는 Hypura가 SSD 수명에 미치는 영향은 사실상 없다.

더 큰 맥락: MoE 스트리밍의 부상

Hypura를 단독 프로젝트로만 볼 게 아니라 더 큰 흐름에서 봐야 한다. HN 댓글에서도 지적됐듯이, 최근 Qwen 3.5 MoE 모델들이 애플 하드웨어에서 "스트리밍 전문가(streaming experts)" 방식으로 인상적인 결과를 보여주고 있다. 1T 파라미터 Kimi K2.5도 실험 대상으로 거론됐다. MoE 아키텍처의 희소 활성화 특성이 NVMe 스트리밍과 본질적으로 궁합이 좋기 때문에, 이 방향의 최적화는 앞으로 더 활발해질 것이다.

지금은 인텔 옵테인(Optane)이 단종된 상태지만, 옵테인의 핵심 가치였던 "RAM과 스토리지 사이의 중간 계층"이라는 콘셉트가 소프트웨어 레벨에서 부활하고 있는 셈이다. 하드웨어가 사라져도 문제의식은 남았고, Hypura 같은 소프트웨어가 그 빈자리를 채우고 있다.

솔직한 고백: AI가 작성한 코드

프로젝트 README 하단에 흥미로운 고백이 있다. 제작자는 "이 저장소의 코드를 직접 작성하지 않았다"고 밝혔다. LLM에게 소크라테스식 질문법으로 방향을 제시하며 코드를 생성시킨 프로젝트라는 것이다. "NVMe를 활용한 추론이 충분히 활용되지 않고 있다"는 직감에서 출발했다고 한다. AI가 만든 AI 추론 최적화 도구라는 점에서 일종의 재귀적 아이러니가 있지만, 기술적 접근법 자체의 타당성과 코드 작성 방식은 별개의 문제다. 결국 아이디어의 가치는 구현보다 설계에 있고, Hypura의 설계 자체는 탄탄하다.

정리

Hypura는 "메모리가 부족하면 포기하라"는 기존 로컬 LLM의 제약에 균열을 낸다. MoE 모델의 희소성을 활용한 전문가 스트리밍은 2.2 tok/s라는 실용적인 속도를 보여줬고, 밀집 모델의 FFN 스트리밍은 느리지만 "실행 불가능"을 "실행 가능"으로 바꿨다. 모델 아키텍처를 이해하는 스토리지 계층 스케줄러라는 접근법은, 컨슈머 하드웨어에서 점점 커지는 모델을 돌리려는 수요가 사라지지 않는 한 계속 발전할 영역이다. 당장 프로덕션에 쓸 도구라기보다는, NVMe를 "느리지만 유효한 메모리 계층"으로 끌어올리는 방향성 자체가 주목할 가치가 있다.

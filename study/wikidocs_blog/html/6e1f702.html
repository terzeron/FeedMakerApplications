AI 기반 지능형 교통신호등 구축 제안서(정보전략실)

  1. 제안 개요
    A 지자체가 추진하는 AI 기반 지능형 교통신호등 설치(총 200개소 중 60개소 변경 설치) 및 기존 신호등 서버/프로그램 전면 폐기 후 신규 구축 사업에 대해, 정보전략실은 다음 목표를 중심으로 전 구간 End-to-End 통합 플랫폼을 제안합니다.

실시간 교통 최적화를 위한 저지연(Edge) + 고도 분석(Center) 하이브리드 구조

AI 모델 자체 개발 계획을 존중하되, 모델 성능·운영(MLOps) 체계까지 제안하여 가점 확보

DBMS 설계 제공 예정이라도, 실시간·시계열·영상 메타데이터를 고려한 확장형 DB 아키텍처 제안으로 가점 확보

60개소 확장 가능성과 200개소 전면 확대까지 고려한 표준화·모듈화 설계

  1. 사업 범위 이해 및 핵심 요구사항 해석
    RFP 요구 기능 정리
    카메라/빗물/밝기 센서 등 다양한 입력

센서 값을 AI 모델로 전달

AI가 최적 교통흐름 유도 신호 예측(결정)

표시부(신호제어기/신호등)에서 결과 표시

우리 제안의 핵심 방향
실시간성이 사업 성패를 좌우
→ 교차로 단위 Edge AI로 1차 판단, 중앙에서 모델 학습/배포/감시

영상 원본은 대역폭/보안 비용이 큼
→ 현장 객체/이벤트 메타데이터 중심 전송, 필요 시 선택적 원본 업로드

  1. H/W 설계(최대한 상세)
    3.1 전체 물리 구성도(권장 아키텍처)
    [교차로(60개소) Edge Zone] ├─ IP Camera(다중) ─┐ ├─ Rain Sensor │ ├─ Luminance Sensor │ ├─ (선택) 레이더/루프/보행감지 │ │ └──> [Edge AI Controller(산업용)] - AI 추론 가속기(NPU/GPU) - 실시간 데이터 수집/정제 - 로컬 캐시/저장 - 신호제어기 연동(I/O, 표준 프로토콜) │ ├──> [신호제어기/표시부] │ - AI 권고 신호 적용/표출 │ └──(전용망/광, 5G/LTE 이중화, VPN) │ [중앙 데이터센터/통합관제 Center Zone] ├─ AI 학습/서빙 GPU 서버(이중화) ├─ 실시간 스트리밍/메시지 서버 ├─ 운영 DB/시계열 DB/데이터레이크 ├─ MLOps/DevOps 플랫폼 ├─ 통합 모니터링/로그/보안관제 └─ 백업/DR(재해복구) 스토리지
    3.2 교차로(Edge) 상세 사양 제안
    1) 센서/입력 장치

IP 카메라

권장: 차량/보행/차로 인식 가능한 해상도 및 야간 성능 확보

설치 방향: 접근로별 최소 1대 이상 구성(교차로 규모 따라 2~6대)

빗물 센서 / 밝기 센서

기상/조도 기반 신호 정책(우천 시 안전 중심, 야간 시 보행 보호 등)

선택형 확장 센서(가점/실효성 강화)

레이더/열화상/보행자 버튼 고도화 등

악천후/야간 인식률 보완

2) Edge AI Controller

산업용 팬리스/내열 설계 권장

구성 요소

고성능 CPU

AI 추론 가속기(NPU/GPU)

1TB 내외 로컬 SSD(캐시/버퍼/장애 대비)

이중 전원/서지 보호

신호제어기 연동 I/F

역할

센서 데이터 수집/시간 동기화

영상 기반 객체 탐지/추적(차량, 보행자, 대기열 길이 등)

초저지연 신호 예측/권고

중앙 시스템 장애 시 로컬 독립 운영(Fallback)

3) 통신

유선 전용망/광 기반 + 무선(5G/LTE) 이중화

Edge-센터 구간

VPN, TLS 적용

QoS 기반 우선순위(제어/메타데이터 우선)

3.3 중앙(Center) 인프라 상세 사양 제안
1) AI 서버군

학습(Training)용 GPU 서버 1~2식

실시간 서빙(Inference/Model Registry) 서버 2식 이중화

모델 버전 관리/검증/롤백 지원

2) 데이터 플랫폼

실시간 스트리밍 서버

운영 DB + 시계열 DB

데이터레이크/오브젝트 스토리지

영상 원본은 정책 기반 선택적 저장

메타데이터(객체수, 속도, 대기열, 이벤트) 중심 저장

3) 운영/보안 인프라

통합 모니터링 서버

로그 수집/분석 서버

취약점/계정/접근 통제 체계

DR/백업

일일 증분 + 주간 전체 백업

핵심 서비스 이중화

  1. S/W 설계(최대한 상세)
    4.1 논리 구성도
    [Edge SW Stack] 1) Device Agent - 카메라/센서 드라이버 - 시간 동기화(NTP/PTP) 2) Data Preprocessor - 노이즈 제거/정규화/이상치 탐지 3) Vision/Signal AI Runtime - 객체 탐지/추적 - 교통상태 추정(속도/대기열/점유율) - 신호 최적화 추론 4) Signal Interface Module - 신호제어기 표준 연동 5) Edge Cache & Store - 장애 시 버퍼링/재전송 6) Security Module - 인증서 기반 통신 [Center SW Stack] A) Data Ingestion - 스트리밍 수집/메시지 큐 B) Real-time Analytics - 교차로/권역 단위 상태 분석 C) AI Platform (MLOps) - 학습 파이프라인 - 모델 레지스트리 - 배포/롤백 D) DB Layer - 운영 DB(구성/자산/정책) - 시계열 DB(센서/이벤트) - 데이터레이크(학습 데이터) E) Operation & Monitoring - 성능/장애/품질 대시보드 F) Admin/Portal - 정책 설정, 리포트, 감사로그
    4.2 데이터 흐름(실시간)
    센서/카메라 데이터 → Edge Agent 수집

Edge에서 전처리 및 객체/교통상태 메타 생성

Edge AI가 1차 신호 권고/결정

결과를 신호제어기에 전달해 표시부 반영

동시에 중앙으로 메타데이터 + 선택 영상 전송

중앙에서 권역 최적화/모델 성능 감시

검증된 모델을 Edge에 안전 배포(카나리/블루그린)

4.3 AI 모델 운용 전략(가점 대응)
지자체의 자체 개발 모델 사용 계획을 기반으로, 당사는 다음을 제안합니다.

모델 공동 고도화 프레임워크

초기 PoC로 60개소 대표 교차로 데이터 확보

자체 모델에 학습 데이터 품질관리, 성능지표 설계 제공

MLOps 도입

학습→검증→배포→모니터링 자동화

모델 편향/성능 저하 감지(Concept Drift)

권장 성능 지표 예시

평균 지체시간 감소율

대기열 길이 감소율

보행자 안전 관련 지표(우천/야간 사고 위험 완화)

추론 지연시간(Edge) 목표치

4.4 DBMS 제안(가점 대응)
발주처가 DB 설계를 제공하더라도, AI 신호 시스템 특성을 반영한 확장형 논리 설계 의견을 제시합니다.

운영/자산/정책 DB

교차로 자산관리(센서/장비/펌웨어 버전)

신호 정책 및 이력

시계열 DB

센서값, 교통상태 지표(속도, 점유율, 대기열)

데이터레이크

AI 학습용 라벨/메타데이터

공간 연계 확장

향후 공간정보본부 데이터와 연동 가능한 구조(교차로 ID 표준화)

  1. 사용 툴(간략 설명)
    5.1 인프라/운영
    Docker: Edge/센터 서비스 컨테이너화로 배포 표준화

Kubernetes(센터): AI/데이터 서비스 오케스트레이션 및 자동 확장

Terraform/Ansible: 인프라 코드 기반 구축·형상 관리

5.2 데이터/스트리밍
Kafka 또는 MQTT: 센서·이벤트 실시간 메시징

ETL/워크플로(예: Airflow): 학습 데이터 정제·적재 자동화

5.3 AI/MLOps
PyTorch/TensorFlow: 모델 개발/학습 프레임워크

MLflow: 실험관리·모델 레지스트리·버전 관리

ONNX/TensorRT 등: Edge 추론 최적화(지연 최소화)

5.4 관제/모니터링/로그
Prometheus + Grafana: 장비/서비스 성능 모니터링

ELK 또는 OpenSearch: 로그 수집·검색·이상 탐지

5.5 DB 후보군(발주처 설계 존중 전제)
PostgreSQL: 운영 데이터

TimescaleDB/InfluxDB: 시계열 데이터

MinIO/S3 호환 스토리지: 학습 데이터/영상 오브젝트 저장

  1. 구축/전환 전략
    6.1 단계적 구축(리스크 최소화)
    설계/표준 수립

교차로 데이터 표준, 장비 규격, 보안 정책 확정

대표 교차로 PoC

3~5개소 우선

성능지표/운영 절차 확정

60개소 확산

권역 단위 순차 적용

안정화/고도화

모델 성능 개선 주기 운영

6.2 기존 시스템 폐기 대응
신규 시스템 전환 시

데이터 이관 필요 여부를 RFP에서 명확화 권고

필요 시 최소 이력만 선별 이관

  1. 보안/안전/신뢰성 설계
    제로트러스트 기반 접속

장비 인증서, 단말 식별, 최소 권한

통신 암호화

Edge↔센터 VPN + TLS

안전(Fail-safe)

중앙 장애 시 Edge 독립 운용

Edge 장애 시 기본 고정 신호 정책 자동 전환

감사/추적

모델 버전별 의사결정 이력 저장

정책 변경 이력 감사로그

  1. 예산(100억) 활용안(예시)
    발주처 세부 산정과 별개로, 정보전략실 관점에서 현장·중앙·운영의 균형 배분을 제안합니다.

Edge 장비/센서/설치(60개소): 45~55%

중앙 AI/데이터센터 인프라: 20~25%

플랫폼 S/W 개발(MLOps/관제/연동): 15~20%

보안·품질·성능검증/PMO: 5~10%

이 구조는 실시간 성능과 지속 운영을 동시에 확보하는 데 유리합니다.

  1. RFP 개선 사항 제안(핵심)
    RFP의 성공적인 집행을 위해 아래 항목의 명시를 권고드립니다.

성능 지표/검증 방법 구체화

지체시간, 대기열, 통행량, 보행 안전지표 등

구축 전후 비교 기간/방법 정의

데이터 범위 및 소유권/보안 정책

안전성 강화

관제 자동화, 장애 예측 정비

MLOps 기반 ‘모델 운영’까지 포함한 완결형 제안

DB/데이터레이크/시계열 통합 설계로 가점 대응

중앙 장애·통신 장애를 고려한 Fail-safe 구조

60개소 구축 후 200개소 확장까지 비용·운영을 선제 설계

부록 B. 간단 요약 슬라이드형 문장(제안서 앞장용)

본 사업은 센서-Edge AI-중앙 MLOps-신호제어가 유기적으로 연결되어야 성공합니다.

당사는 저지연 Edge 기반 실시간 제어와 중앙 AI/데이터 플랫폼을 결합한 하이브리드 아키텍처를 제안합니다.

지자체의 자체 AI 개발 방침을 존중하면서도, 모델 성능의 지속적 검증·배포·감시 체계(MLOps)를 제공하여 실효성과 가점을 동시에 확보하겠습니다.

또한 운영 DB, 시계열 DB, 학습 데이터레이크를 연계한 확장형 DB 아키텍처를 제안하여 향후 200개소 확대에도 안정적으로 대응하겠습니다.
